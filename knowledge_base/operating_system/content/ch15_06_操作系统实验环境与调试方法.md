# 操作系统实验环境与调试方法

## 实验环境的目标

操作系统实验的价值在于把抽象概念落实到代码和工具上。只读教材容易理解概念名词，但真正实现 shell、内存分配器、线程库、文件系统检查器或 xv6 系统调用时，才能看到边界条件和错误路径。

实验环境通常基于 UNIX、C、编译器、调试器、make、系统调用文档和测试脚本。掌握这些工具是做系统项目的前提。

## 编译工具链

C 程序从源文件到可执行文件通常经历预处理、编译、汇编和链接。`gcc` 或类似编译器驱动会协调这些步骤。常用选项包括 `-Wall` 打开警告，`-g` 生成调试信息，`-O` 开启优化，`-o` 指定输出文件。

警告不应忽略。系统代码里一个类型不匹配、未初始化变量或错误返回值处理，都可能变成难以复现的崩溃。实验代码应从一开始就保持干净编译。

## Makefile

项目变大后，应使用 Makefile 管理编译依赖。单独编译源文件，再链接成可执行文件，可以避免每次改动都重编全部代码。Makefile 还可统一测试命令、清理命令和编译选项。

好的实验工程应让构建过程可重复。评测脚本和同伴都应能用同一命令得到同样结果。

## 调试方法

调试系统程序要结合多种工具。`gdb` 用于断点、单步、查看栈和变量；`strace` 可观察系统调用；日志输出可记录并发时序；十六进制查看工具可检查磁盘映像和二进制结构。

调试时应先缩小输入，再固定随机性，最后验证假设。并发和文件系统错误尤其需要可重复测试，否则只靠观察偶发现象很难定位。

## 测试意识

系统实验不能只测正常路径。shell 要测重定向、管道、错误命令和子进程回收；内存分配器要测碎片、重复释放、边界大小；文件系统检查器要测坏指针、重复块、链接计数和目录格式。

自动化测试能防止回归。每修一个问题，都应留下对应测试或至少记录可复现步骤。操作系统代码的可靠性来自大量边界条件验证。

# 程序、进程与执行上下文

## 程序与进程的区别

程序是静态对象，通常以可执行文件、脚本或字节码的形式保存在磁盘中。它包含指令、初始数据、符号信息、动态链接需求等内容，但在没有被装入并执行之前，它不会消耗 CPU，也没有运行时状态。进程是程序运行起来之后形成的动态对象。操作系统为它分配地址空间、建立内核记录、安排调度，并把 CPU 在某一时刻的执行状态与它绑定起来。

因此，“一个程序”与“一个进程”不是一一对应关系。同一个程序可以被启动多次，形成多个独立进程；一个进程也可以在执行 `exec` 后把自己的程序映像替换成另一个可执行文件。真正决定进程身份的不是可执行文件名，而是内核维护的进程实体以及它拥有的一组运行时资源。

## 进程的机器状态

理解进程必须看它的机器状态。机器状态指程序继续执行所需的全部关键信息，至少包括地址空间、通用寄存器、程序计数器、栈指针、打开文件表项、信号状态以及调度相关信息。地址空间保存代码、数据、堆、栈和内存映射区域；寄存器保存当前计算的中间值；程序计数器指向下一条将要执行的指令；栈指针定位当前函数调用链。

这些状态让操作系统可以暂停一个进程，再在之后恢复它。暂停时，内核需要把处理器上的寄存器内容保存到该进程的内核数据结构中；恢复时，再把这些值装回处理器。只要保存和恢复足够完整，进程就会像“从未离开 CPU”一样继续运行，这正是 CPU 虚拟化能够成立的基础。

## 执行上下文包含什么

执行上下文比“寄存器现场”更宽。狭义上下文指一次上下文切换中必须保存的硬件执行现场，例如 PC、SP、通用寄存器和状态寄存器。广义上下文还包括地址空间映射、内核栈、进程状态、优先级、所属用户、资源限制、当前工作目录、打开文件描述符、信号处理设置等。

用户态程序只能直接看到自己地址空间中的数据和部分可查询属性；内核态则必须掌握完整上下文，因为它要在调度、系统调用、异常、中断和进程退出时做一致性维护。例如，一个进程调用 `read()` 阻塞等待磁盘 I/O 时，内核不仅要保存它的 CPU 现场，还要把它挂入对应设备的等待队列，并把状态从运行改为阻塞。

## 从程序到进程

创建进程时，操作系统通常要完成几类工作。第一，读取可执行文件头，确定代码段、数据段、入口地址和动态链接信息。第二，建立地址空间，把程序需要的区域映射到虚拟内存中；现代系统常常采用按需装入，只有真正访问到某页时才从文件或零页创建物理页。第三，初始化用户栈，把命令行参数、环境变量和辅助信息放到约定位置。第四，建立内核侧的进程记录，使调度器、文件系统、信号机制和内存管理模块能够找到它。

最后，内核把新进程的初始寄存器设置成“即将从程序入口开始执行”的样子。当调度器选择它运行时，CPU 控制权转入用户态，程序才真正开始作为进程执行。

## 易错点

不要把进程理解成“正在占用 CPU 的程序”。一个进程即使当前没有运行，只要它的状态和资源仍由内核维护，它仍然存在。就绪进程等待 CPU，阻塞进程等待事件，僵尸进程已经执行结束但退出状态尚未被父进程回收。

也不要把上下文只理解为内存内容。进程的连续性依赖寄存器、页表、内核栈、文件描述符和调度状态共同维持。缺少其中任何关键部分，操作系统就无法正确恢复执行。

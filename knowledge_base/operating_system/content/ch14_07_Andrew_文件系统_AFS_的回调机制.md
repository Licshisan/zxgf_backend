# Andrew 文件系统 AFS 的回调机制

## AFS 的扩展目标

AFS 关注大规模客户端环境下的可扩展性。若每个客户端频繁向服务器验证缓存，服务器 CPU 和网络带宽会成为瓶颈。AFS 的设计目标是让客户端在缓存有效时尽量本地访问，减少服务器交互。

AFS 使用全文件缓存。客户端首次打开文件时从服务器获取整个文件，保存到本地磁盘；之后读写走本地副本；关闭时若文件被修改，再把整个文件写回服务器。

## AFSv1 的问题

早期 AFS 会在访问缓存文件时向服务器发送验证请求，还会把完整路径名传给服务器，让服务器执行路径遍历。测量显示，服务器花费大量时间处理路径查找和缓存验证。

这些问题限制扩展性。客户端越多，服务器越忙于回答“这个缓存还有效吗”以及重复路径解析，而不是服务真正的数据传输。

## 回调

AFSv2 引入回调。服务器在向客户端返回文件或目录时，同时承诺：如果该对象被其他客户端修改，服务器会通知持有缓存的客户端，使其缓存失效。客户端在未收到失效通知前，可以相信本地缓存有效。

回调类似把轮询改成通知。NFS 客户端需要定期检查服务器，AFS 客户端则默认使用缓存，直到服务器打破承诺。这样显著减少缓存验证流量。

## 文件标识符

AFS 还使用文件标识符 FID 替代完整路径名。FID 包含卷、文件编号和唯一标识信息。客户端逐级查找并缓存目录结果，后续访问可以直接用 FID 与服务器交互。

这减少了服务器重复路径遍历负担。路径名解析的一部分工作转移到客户端缓存中，服务器只需处理更直接的对象请求。

## 一致性语义

跨机器时，AFS 的更新通常在关闭文件时对服务器可见。服务器收到新版本后，撤销其他客户端相关回调。其他客户端下一次打开该文件时，会重新获取新版本。

同一客户端机器上的进程则通常能立即看到彼此对本地缓存文件的修改，更接近本地 UNIX 语义。这个模型清晰，但并不解决多个客户端同时写同一文件的应用级冲突。

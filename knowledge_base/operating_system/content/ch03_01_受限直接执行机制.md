# 受限直接执行机制

## 基本思想

CPU 虚拟化要同时满足两个目标：程序运行要快，操作系统又必须保留控制权。受限直接执行就是围绕这两个目标形成的机制。所谓“直接执行”，是让用户程序的大多数指令直接在真实 CPU 上运行，而不是由操作系统逐条解释；所谓“受限”，是通过硬件特权级、陷入机制和定时器中断限制程序能做什么、能连续运行多久。

如果只追求速度，操作系统可以把程序加载到内存后直接跳到入口点执行。但这样程序可能执行特权指令、访问设备、修改页表或永远占用 CPU。受限直接执行的核心价值在于：普通计算路径尽量不打扰应用，涉及权限、资源和调度的关键点必须回到内核。

## 直接执行的准备工作

启动一个进程前，内核需要创建进程记录，建立地址空间，装入或映射可执行文件，初始化用户栈，并设置初始寄存器。程序计数器被设置为用户程序入口，栈中放入参数和环境变量。随后内核通过特定返回路径切换到用户态，CPU 开始执行应用指令。

一旦进入用户态，应用的普通算术、分支、内存访问等操作都直接由硬件完成。操作系统不会参与每条指令，否则性能无法接受。只有当应用需要服务、发生异常、收到中断或被抢占时，控制权才回到内核。

## “受限”来自哪里

限制首先来自处理器模式。用户态代码不能执行特权指令，不能任意访问内核内存，不能直接操作设备控制寄存器，也不能随意修改中断表或页表。若用户程序尝试做这些事，硬件会触发异常，把控制权交给内核处理。

限制还来自系统调用接口。应用可以通过 `read()`、`write()`、`fork()` 等受控入口请求内核服务，但不能指定跳转到内核任意地址。内核在启动时设置陷入表，硬件只允许系统调用进入预先登记的处理程序。这样既给应用提供必要能力，又避免应用绕过权限检查。

## 控制权问题

受限直接执行必须解决“内核如何重新获得 CPU”这一问题。协作式方案依赖进程主动系统调用或让出 CPU，但恶意或错误程序可能无限循环。现代通用系统通常使用定时器中断：内核配置硬件定期产生中断，即使用户程序不主动调用系统调用，CPU 也会在中断到来时进入内核。

定时器中断让操作系统具备抢占能力。内核可以检查当前进程是否用完时间片，决定继续运行它还是切换到其他进程。没有这个机制，CPU 虚拟化就缺少可靠的时间复用基础。

## 机制与策略的边界

受限直接执行本身是机制。它回答如何安全地运行用户代码、如何进入内核、如何返回用户态、如何抢占当前进程。至于抢占后选择哪个进程运行、时间片设多长、优先级如何调整，则属于调度策略。

把机制和策略分开很重要。硬件陷入、特权级和上下文切换为所有调度器提供共同基础；不同调度算法可以在此基础上追求低延迟、高吞吐、公平性或实时性。

## 易错点

受限直接执行不是“让操作系统一直监督程序”，而是让程序大部分时间直接运行，只在必要边界进入内核。它也不是单纯的软件技巧，必须依赖硬件支持，包括特权级、陷入指令、中断、异常和受保护的地址空间。

# shell、重定向与管道的进程组合

## shell 的进程管理角色

shell 本身是普通用户程序，但它通过系统调用组织其他程序运行。它读取用户输入，解析命令、参数、重定向和管道，然后使用 `fork()`、`exec()`、`wait()`、`pipe()`、`dup2()`、`close()` 等接口构造进程组合。操作系统提供机制，shell 决定如何把这些机制组合成用户可见的命令语言。

最简单的前台命令流程是：shell 创建子进程；子进程调用 `exec()` 变成目标程序；父进程调用 `wait()` 等待该命令结束；结束后 shell 再显示提示符。后台命令则不同，shell 不等待或只记录任务信息，让用户继续输入下一条命令。

## 重定向如何工作

重定向依赖文件描述符继承。UNIX 约定 0 是标准输入，1 是标准输出，2 是标准错误。执行 `cmd > out.txt` 时，shell 在子进程中打开 `out.txt`，把文件描述符 1 指向该文件，再执行 `cmd`。目标程序仍然调用 `printf()` 或 `write(1, ...)`，但数据已经流向文件而不是终端。

输入重定向类似。执行 `cmd < in.txt` 时，子进程让描述符 0 指向输入文件，目标程序照常从标准输入读取。错误重定向则操作描述符 2。由于这些修改发生在 `fork()` 之后、`exec()` 之前，不会破坏 shell 自己的标准输入输出。

## 管道如何连接进程

管道提供单向字节流，常用于把一个进程的输出连接到另一个进程的输入。执行 `producer | consumer` 时，shell 创建管道，得到读端和写端两个文件描述符；第一个子进程把标准输出复制到写端，关闭不用的读端；第二个子进程把标准输入复制到读端，关闭不用的写端；随后二者分别 `exec()` 成目标程序。

管道的关键语义是读写阻塞和 EOF。读端在没有数据但写端仍打开时会阻塞；当所有写端关闭后，读端读到 EOF。若 shell 或某个子进程忘记关闭不需要的管道端，消费者可能永远等不到 EOF，表现为命令卡住。

## 多段管道与并发执行

多段管道如 `cat file | grep key | wc -l` 会创建多个进程和多个管道。它们通常并发运行，而不是前一个完全结束后再启动后一个。这样可以流式处理数据，减少中间文件和内存占用。调度器在这些进程之间切换，管道缓冲区负责吸收短暂的速度差异。

如果生产者写得太快而管道缓冲区已满，写操作会阻塞；如果消费者读得太快而暂无数据，读操作会阻塞。这种背压机制让进程组合在不显式共享内存的情况下协同工作。

## shell 为什么需要 fork 后 exec 前的窗口

`fork()` 和 `exec()` 分离的设计给 shell 留出了改造子进程环境的窗口。重定向、管道、设置环境变量、改变工作目录、调整进程组、恢复默认信号处理，都可以在子进程中完成，然后再 `exec()`。如果只有“直接创建并运行某程序”的单一调用，shell 要实现这些功能会困难得多。

这也是 UNIX API 设计的一个重要思想：简单机制可以通过组合表达复杂行为。`fork` 负责复制执行环境，文件描述符机制负责统一文件、终端和管道，`exec` 负责替换程序映像，`wait` 负责回收与同步。

## 易错点

重定向和管道不是目标程序的特殊能力，而是 shell 和内核文件描述符机制共同提供的效果。大多数命令只知道自己在读标准输入、写标准输出。另一个常见错误是忘记关闭多余描述符：管道 EOF、资源释放和安全边界都依赖正确关闭。分析 shell 行为时，应始终区分父 shell 的描述符和子进程继承后的描述符。

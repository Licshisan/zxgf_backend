# fork 系统调用与地址空间复制

## fork 的基本语义

`fork()` 是 UNIX 进程创建模型中的核心系统调用。调用成功后，系统中出现两个进程：原来的父进程和新创建的子进程。子进程几乎继承父进程的执行环境，包括地址空间内容、打开文件描述符、当前工作目录、环境变量、信号设置和资源限制。二者都从 `fork()` 返回后的同一位置继续执行。

`fork()` 最特别的地方在返回值。父进程得到子进程的 PID，子进程得到 0；如果创建失败，父进程得到负值或错误码。程序正是依靠这个差异区分父子路径。子进程不是从 `main()` 重新开始，也不会重新执行 `fork()` 之前的代码，这一点对理解输出顺序和变量状态非常重要。

## 地址空间看似复制

从语义上说，子进程获得父进程地址空间的一份副本。`fork()` 前内存中的变量值在子进程中可见，但父子进程之后对变量的修改互不影响。原因是它们拥有独立的虚拟地址空间：同一个虚拟地址在父子进程中可以映射到不同物理页。

如果真的在 `fork()` 时立即复制全部物理内存，成本会非常高，尤其是父进程很大而子进程马上执行 `exec()` 的场景。现代系统通常使用写时复制。父子进程最初共享只读物理页，页表项被标记为不可写；当任一方尝试写入时，处理器触发缺页异常，内核再复制对应页面并恢复写权限。这样只复制实际被修改的页。

## 继承的不只是内存

`fork()` 还会复制或共享许多非内存资源。文件描述符会在子进程中保留，但它们通常指向同一打开文件对象，因此父子进程可能共享文件偏移量和打开状态。比如父进程打开一个文件后 `fork()`，父子各自写入时，写入顺序取决于调度，文件偏移也会共同前进。

信号处理、当前工作目录、环境变量等也会继承。进程 ID、父进程 ID、资源统计、未决信号等则不会简单照搬。理解哪些资源复制、哪些资源共享、哪些资源重新生成，是分析 `fork()` 行为的关键。

## fork 后的不确定顺序

`fork()` 返回后，父进程和子进程都处于可运行状态，谁先执行由调度器决定。单 CPU 上二者交替运行，多 CPU 上还可能真正并行。因此，除非使用 `wait()`、管道、锁或其他同步机制，程序不能假设父进程一定先输出，或子进程一定先完成。

这种不确定性不是错误，而是并发系统的正常属性。正确程序应当只依赖明确同步建立的顺序。例如父进程需要读取子进程产生的文件时，应等待子进程结束或通过 IPC 确认写入完成，而不是靠“通常子进程会很快执行”这样的经验。

## 常见用途

`fork()` 常与 `exec()` 配合使用。shell 读取命令后先 `fork()` 出子进程，在子进程中设置重定向、管道和环境，然后调用 `exec()` 执行目标程序；父进程则根据前台或后台任务决定是否 `wait()`。这种拆分让 shell 有机会在新程序运行前改造其执行环境。

服务器程序也可能使用 `fork()` 为连接创建子进程，以隔离请求处理逻辑。虽然现代高性能服务器更多使用线程、事件循环或进程池，但 `fork()` 仍是理解 UNIX 进程模型的基础。

## 易错点

`fork()` 不等于启动一个全新程序，它只是复制当前进程的执行环境。要运行不同程序，需要后续调用 `exec()`。另外，写时复制是实现优化，不改变语义；即使物理页暂时共享，父子进程仍应被看作拥有独立地址空间。最后，`fork()` 后多了一条执行流，未同步访问共享文件描述符、终端输出或外部文件时，结果顺序可能变化。

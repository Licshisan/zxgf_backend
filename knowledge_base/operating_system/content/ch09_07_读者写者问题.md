# 读者写者问题

## 问题定义

读者写者问题描述一类共享对象：多个线程可以同时读取，但写入必须独占。读操作之间不互相破坏，写操作会改变状态，因此写者不能与任何读者或其他写者并发执行。文件缓存、配置表、路由表、内存索引都可能符合这种访问模式。

普通互斥锁能保证正确性，但会把多个读者也串行化，浪费读多写少场景中的并发潜力。读写锁通过区分读锁和写锁提升并发度。

## 基本语义

线程获取读锁后，只要没有写者持锁，其他读者也可以进入。线程获取写锁时，必须等待所有读者和写者离开；写者持锁期间，新的读者和写者都不能进入。

读写锁保护的是同一组共享状态。读者不能修改被保护数据，否则会破坏“多个读者可并发”的前提。

## 公平性策略

读者优先策略让新读者在已有读者活动时继续进入，吞吐高，但写者可能长期饥饿。写者优先策略在写者等待时阻止新读者进入，保证写者能尽快执行，但可能降低读吞吐。公平策略按到达顺序排队，避免两边饥饿。

实际系统要根据负载选择策略。读多写少且写延迟不敏感时，读者优先可能合适；配置更新、元数据修改等写操作不能长期等待时，应避免读者无限插队。

## 实现要点

读写锁通常维护活动读者数、是否有活动写者、等待写者数以及条件变量或信号量。读者进入时检查是否有写者活动，可能还要检查是否有等待写者。最后一个读者离开时，若有写者等待，需要唤醒写者。

写者进入时等待活动读者数为 0 且没有活动写者。写者离开后，根据策略唤醒一个写者或广播给多个读者。

## 易错点

读写锁不是总比互斥锁快。读临界区很短、竞争低或写很多时，读写锁的计数和唤醒开销可能更大。读锁内不能执行写操作，也不能在持读锁时盲目升级为写锁，否则容易死锁。

# 文件抽象与字节流模型

## 文件是什么

在 UNIX 风格系统中，文件的基本抽象是一段持久保存的线性字节序列。操作系统通常不解释这些字节的内部格式：它不关心文件是 C 源码、图片、数据库页还是日志记录。文件系统的职责是保存这些字节，并在之后按请求返回同样的内容。

这种抽象把复杂的持久存储设备变成了程序容易使用的对象。应用不必知道数据位于哪块磁盘、哪个扇区、是否经过缓存或 RAID 控制器，只需要用文件名打开文件，并通过读写接口访问字节范围。

## 低级名称与用户名称

文件在文件系统内部有低级名称，UNIX 中通常是 inode 号。用户平时使用的是路径名，例如 `/home/user/a.txt`。目录负责把用户可读名称映射到低级名称，因此文件名不是文件本体的一部分，而是目录项对 inode 的引用。

同一个文件可以有多个名字，这就是硬链接的基础。反过来，删除一个名字也不一定立即删除文件内容，只有当目录引用和打开引用都消失后，文件系统才可以回收其 inode 和数据块。

## 字节流模型

字节流模型意味着文件可以看成从偏移 0 开始的一维数组。`read` 从某个偏移读取若干字节，`write` 从某个偏移写入若干字节，文件大小随写入扩展。文件系统不会自动理解记录边界，应用若需要“第几条记录”这样的结构，必须自己在字节流上定义格式。

字节流带来高度通用性。文本文件用换行分隔，数据库用固定大小页，压缩文件用自己的头部和块结构，所有这些都能放进同一套文件接口中。代价是语义主要由应用承担，文件系统只提供字节级保存和命名。

## 持久性与缓存

文件是持久存储抽象，但写入接口返回成功不总是表示数据已经到达稳定介质。多数文件系统会把数据先放入页缓存或缓冲区缓存，稍后批量写回磁盘，以改善性能。应用若需要明确的持久化边界，应使用 `fsync`、同步打开标志或数据库自己的日志协议。

因此，“文件是持久的”应理解为文件系统最终负责把内容保存到持久设备，而不是每次 `write` 都立即落盘。理解这一区别对崩溃一致性非常重要。

## 抽象的边界

文件抽象隐藏设备细节，但不保证所有操作都相同。普通文件支持随机访问；管道和套接字也使用文件描述符接口，却通常不支持 `lseek`；设备文件可能把 `read` 和 `write` 映射到硬件行为。UNIX 的统一接口提高了组合性，但具体对象仍有不同语义。

学习文件抽象时，要区分三个层次：路径名如何找到文件，打开文件对象如何记录访问状态，底层文件系统如何把字节映射到磁盘块。混淆这些层次会导致对链接、偏移量和持久化语义的误解。

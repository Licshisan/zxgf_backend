# 早期系统、多道程序与时分共享

## 早期内存模型

早期计算机系统几乎没有内存抽象。操作系统占据物理内存的一部分，用户程序占据另一部分，程序直接使用物理地址。系统一次通常只运行一个用户程序，保护需求较弱，内存管理也相对简单。

这种模型的问题也很明显。程序如果写错地址，可能破坏操作系统或其他数据；程序必须适应实际加载位置；机器资源昂贵却经常空闲。当用户需求从单一批处理转向多用户、交互和更高利用率时，简单物理内存模型无法满足要求。

## 多道程序

多道程序系统让多个进程同时驻留在内存中。当一个进程等待 I/O 时，操作系统可以切换到另一个进程运行，提高 CPU 利用率。这样，磁盘、终端和 CPU 能更好地重叠工作，不必让昂贵 CPU 在 I/O 等待期间空转。

多道程序要求操作系统能区分不同进程的内存区域。否则一个进程的错误写入可能破坏另一个进程，使系统可靠性大幅下降。保护和隔离因此成为内存虚拟化的重要动机。

## 时分共享

时分共享进一步强调交互性。多个用户通过终端同时使用同一台机器，操作系统把 CPU 时间切成片段，在进程之间快速切换，让每个用户感觉自己正在独占机器。CPU 虚拟化和内存虚拟化在这里结合：CPU 需要上下文切换，内存需要让多个进程长期驻留而互不干扰。

一种朴素做法是在切换进程时把整个内存映像写到磁盘，再装入另一个进程。但这极慢，内存越大越不可行。更合理的做法是让多个进程同时留在内存中，只切换寄存器、页表或重定位信息。

## 新需求

多道程序和时分共享带来几个关键需求。第一，重定位：进程不应必须加载到固定物理地址。第二，保护：进程只能访问自己的内存。第三，效率：切换进程时不能复制整个地址空间。第四，扩展性：系统应支持多个进程、可变大小程序和不同内存需求。

动态重定位、分段、分页和后来的虚拟内存机制都在回应这些需求。它们不是凭空出现的复杂设计，而是共享机器、保护进程和提高利用率的结果。

## 易错点

多道程序不是多处理器。即使只有一个 CPU，也可以让多个进程驻留内存并交替执行。时分共享也不等于真正并行，它依靠快速切换制造并发体验。理解这些历史背景，有助于理解为什么操作系统要引入地址空间和硬件地址转换。

# 轮询、中断与 DMA

## 轮询

轮询是 CPU 反复读取设备状态寄存器，直到设备就绪或完成。它的优点是实现简单、延迟可控，不需要中断上下文切换；缺点是等待期间 CPU 被浪费，设备越慢浪费越明显。

轮询适合非常快的设备、预计很快完成的短操作，或系统希望主动控制处理节奏的场景。网络系统在高负载下有时会采用轮询或混合策略，避免每个数据包都触发中断导致 CPU 长时间陷在中断处理里。

## 中断

中断让设备在完成 I/O 后主动通知 CPU。内核提交请求后可让当前进程睡眠并调度其他进程；设备完成时触发中断处理程序，处理程序读取完成状态、回收请求、唤醒等待者。这样 CPU 计算和设备 I/O 可以重叠，提高整体利用率。

中断并非无代价。进入中断处理、保存恢复上下文、调度唤醒线程都需要时间。如果设备完成极快，中断开销可能大于轮询；如果中断频率过高，系统可能忙于处理中断而没有时间执行用户程序。为降低开销，设备和驱动常使用中断合并，在短时间内把多个完成事件合成一次通知。

## DMA 的必要性

如果所有数据都由 CPU 逐字节或逐字搬运到设备寄存器，这种编程 I/O 会占用大量处理器时间。DMA 允许设备控制器或 DMA 引擎直接在内存和设备之间传输数据。CPU 只需设置源地址、目标地址、长度和方向，然后启动传输。

DMA 的收益是把 CPU 从大块数据搬运中释放出来，使它可以运行其他线程。磁盘、网卡、声卡和显卡等高吞吐设备普遍依赖 DMA，否则 CPU 会成为 I/O 通路的瓶颈。

## DMA 的正确性问题

DMA 不是简单的“设备自己拷贝”。操作系统必须保证 DMA 缓冲区在传输期间有效，不能被释放、换出或重映射；还要处理缓存一致性，确保设备看到的是 CPU 已写好的数据，CPU 读取的是设备已经写完的新数据。某些系统需要刷新或失效缓存，某些系统通过一致性硬件完成。

为了限制设备访问范围，现代系统常使用 IOMMU。它为设备提供类似页表的地址转换和权限控制，防止故障设备或恶意设备任意读写物理内存。没有这种保护时，DMA 能绕过 CPU 的普通内存保护机制。

## 混合策略

实践中常把轮询、中断和 DMA 组合使用。驱动可能先短暂轮询，如果设备未完成再睡眠等待中断；大块数据用 DMA，小型控制命令用寄存器写入；高负载网络用轮询批量取包，低负载时用中断降低空转。

选择哪种机制取决于设备延迟、请求大小、负载强度和系统目标。低延迟系统倾向减少睡眠唤醒；吞吐系统倾向批处理和 DMA；通用操作系统则需要在 CPU 利用率、响应时间和实现复杂度之间平衡。

# 比较并交换 CAS 与无锁基础

## CAS 的语义

比较并交换 CAS 是常见原子指令。它读取某地址当前值，与期望值比较；若相等，就把该地址更新为新值并报告成功；若不相等，不修改并报告失败。整个比较和写入过程原子执行。

CAS 可以表达“只有状态仍是我观察到的样子，才提交修改”。这使它成为无锁算法和许多同步原语的基础。

## 用 CAS 实现锁

锁初始为 0。线程执行 `CAS(&lock, 0, 1)`，成功则获得锁，失败则继续等待。释放时写回 0。与 TAS 相比，CAS 能区分期望状态和目标状态，也能构建更复杂的数据结构更新。

但简单 CAS 自旋锁仍有自旋锁的竞争和公平性问题。CAS 是工具，不是自动高性能方案。

## 无锁数据结构

无锁算法通常用 CAS 在循环中尝试提交更新。例如无锁栈入栈时，线程读取旧栈顶，把新节点的 next 指向旧栈顶，然后 CAS 把栈顶从旧值改为新节点。若失败，说明其他线程已经修改栈顶，重新读取并重试。

无锁算法的好处是某个线程暂停或崩溃不一定阻塞所有线程。代价是实现复杂，重试可能很多，内存回收尤其困难。

## ABA 问题

CAS 只比较值是否等于期望值。如果一个指针从 A 变为 B 又变回 A，CAS 会以为状态没有变化，但实际对象可能已经被删除并重新分配。这就是 ABA 问题。

解决方法包括版本号标记、双宽 CAS、危险指针、epoch 回收等。没有安全内存回收，无锁链表、栈和队列很容易出错。

## 内存顺序

CAS 和其他原子操作还涉及内存顺序。不同平台可能允许读写重排序。正确无锁算法必须指定 acquire、release、seq_cst 等语义，确保数据初始化、指针发布和后续读取顺序正确。

这也是为什么无锁代码难写。仅仅“用了 CAS”并不意味着并发正确。

## 易错点

无锁不等于无等待。某些线程可能反复 CAS 失败而饥饿。无锁也不一定更快，低竞争下普通锁可能更简单更高效。CAS 适合构建原子状态转换，但复杂共享结构仍需严谨证明。

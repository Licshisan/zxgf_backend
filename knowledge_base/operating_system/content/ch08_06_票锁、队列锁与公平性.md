# 票锁、队列锁与公平性

## 简单自旋锁的公平问题

TAS 或 CAS 自旋锁释放时，所有等待线程竞争同一个锁变量。谁先抢到取决于调度、缓存位置和硬件时序，不保证先等待者先获得。竞争激烈时，某些线程可能长期失败。

公平锁试图让等待顺序更可控，同时降低共享缓存行上的争用。

## 票锁

票锁维护两个计数：下一个发号值和当前服务号。线程获取锁时原子获取一个票号，然后自旋等待当前服务号等于自己的票号。释放锁时，持有者把服务号加一。

票锁提供 FIFO 公平性。每个线程按取票顺序进入临界区，不会被后来者插队。它实现相对简单，适合需要公平的自旋场景。

## 票锁的代价

所有等待线程都在读同一个服务号。释放锁时服务号更新，会让多个 CPU 的缓存行同时失效，等待线程一起重新读取。在大规模多核系统中，这种共享热点仍会造成通信开销。

票锁公平，但不一定最具扩展性。

## 队列锁

队列锁把等待线程组织成队列，每个线程通常在自己的本地节点或本地标志上自旋。释放锁时，持有者只唤醒后继节点。MCS 锁和 CLH 锁是经典队列锁。

队列锁减少了所有线程竞争同一缓存行的问题，扩展性更好，也天然提供等待顺序。它更适合多核高竞争环境。

## 选择取舍

简单自旋锁代码短，无竞争时快；票锁公平但共享服务号热点明显；队列锁扩展性好但实现复杂，每个等待者需要节点管理。内核和运行时会根据临界区长度、CPU 数、竞争程度选择实现。

## 易错点

公平性不等于性能最好。严格 FIFO 可能让刚释放锁的 CPU 无法利用缓存优势，也可能增加延迟。队列锁减少争用，但仍是自旋等待，持锁线程若被抢占，后续线程仍会浪费 CPU。

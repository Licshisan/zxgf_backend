# I/O 设备模型与总线结构

## 为什么 I/O 需要单独建模

CPU 和内存的速度通常远高于外设，而外设又种类繁多：磁盘、网卡、显示设备、键盘、USB 存储和传感器的延迟、带宽、错误模式都不同。操作系统不能为每种设备在上层重写一套文件系统或应用接口，因此需要把设备抽象成可提交请求、可报告状态、可传输数据的受控对象。

I/O 设备模型的核心不是“设备很慢”这一句，而是系统如何在异构硬件之间建立统一边界：上层只看到读、写、控制、等待完成等操作；底层由控制器、寄存器、总线、DMA 和驱动程序处理具体协议。

## 分层总线结构

典型机器采用分层连接。CPU 通过高速互连访问内存，显卡、网卡等高性能设备连接在更靠近处理器的 I/O 总线上，磁盘、鼠标、键盘、U 盘等较慢设备连接在外围总线上。这样设计的原因包括电气距离、布线成本、带宽需求和设备数量。

高速总线通常更短、更贵、可连接设备数量更少；低速外围总线延迟更高但扩展性更好。操作系统在设计 I/O 子系统时必须理解这种层次，因为一次设备访问可能跨越多个桥接器和控制器，延迟不只来自设备本体，也来自总线仲裁、数据搬运和中断传递。

## 设备接口与内部结构

一个设备通常由两部分构成：对主机暴露的接口和设备自身的内部实现。接口包括寄存器、命令格式、状态位、错误码以及数据传输方式。内部实现则可能只是简单电路，也可能包含处理器、固件、缓存和专用逻辑。例如 RAID 控制器本身就像一台小型计算机，负责把主机请求拆成多个磁盘请求。

这种分离带来一个重要结论：操作系统并不需要知道设备内部如何完成磁头移动、纠错或缓存预取，但必须严格遵守设备接口协议。违反协议，例如在设备忙时重复提交命令，可能导致请求丢失、数据损坏或控制器复位。

## I/O 访问的基本路径

应用程序通常通过系统调用发起 I/O，请求进入内核后被文件系统、块层或网络协议栈转换为设备请求。驱动程序再把请求翻译成设备能理解的寄存器写入、命令描述符或 DMA 描述符。设备完成后通过中断、状态位或完成队列通知内核，内核再更新等待队列、缓冲区状态和用户可见的返回值。

这条路径说明 I/O 是跨层协作：应用关心语义，内核关心权限和调度，驱动关心硬件协议，设备关心执行命令。任何一层过度暴露细节都会增加耦合，任何一层过度隐藏错误又会削弱恢复能力。

## 设计取舍

I/O 子系统追求三类目标：通用性、性能和可靠性。通用性要求文件系统可以运行在不同磁盘之上；性能要求 CPU 等待设备时能去执行其他任务，并尽量利用 DMA、批处理和请求合并；可靠性要求系统处理设备错误、超时、中断丢失和部分完成。

总线结构和设备模型不是孤立知识点。它们直接影响驱动程序如何写、磁盘调度是否有效、文件系统缓存如何组织，以及崩溃一致性机制能否真正把数据安全落到稳定存储。

# 有序写入、写屏障与崩溃窗口

## 写入顺序为何重要

文件系统更新常包含依赖关系。数据块应先于指向它的 inode 持久化，日志事务内容应先于提交记录持久化，目录项不应先于对应 inode 初始化完成。若顺序被打乱，崩溃后可能出现合法元数据指向错误内容。

应用调用系统调用的顺序不一定等于设备落盘顺序。内核缓存、I/O 调度器、磁盘控制器和写缓存都可能重排请求，以提高性能。

## 有序写入

有序写入要求某些写入在另一些写入前完成。例如元数据日志的有序模式中，用户数据块必须先写入最终位置，然后才能提交引用这些块的元数据事务。这样崩溃后即使文件大小变大，也不会读到未初始化块。

有序写入不等于同步写入所有数据。它只约束关键依赖，允许其他无关写入继续缓存和重排。好的文件系统会尽量只为必要顺序付出代价。

## 写屏障与刷新

写屏障用于阻止屏障前后的写入跨越重排。刷新命令要求设备把易失性缓存中的数据写入稳定介质。文件系统在提交记录、fsync、日志回收等位置需要这些机制。

如果设备谎称刷新完成，或控制器缓存没有掉电保护，文件系统再正确也可能失去持久化保证。因此企业存储常使用带电池或电容保护的缓存，并要求设备正确实现 flush 语义。

## 崩溃窗口

崩溃窗口是某个操作开始后到其持久化保证成立前的时间段。在普通写回模式下，窗口可能持续数秒；在 fsync 返回前，窗口仍未关闭；在日志提交记录落盘后，事务的恢复保证才成立。

缩小崩溃窗口通常需要更频繁同步，代价是吞吐下降和延迟升高。文件系统和应用必须根据数据重要性选择窗口大小，而不是默认认为所有写入都立即安全。

## 应用层影响

数据库、编辑器和包管理器常用临时文件、fsync 和 rename 构造可靠更新。若漏掉目录 fsync，可能出现文件内容已落盘但新名字未持久化的情况。若只依赖 rename 原子性而不 fsync 文件，可能原子地替换成空文件或旧内容。

持久化语义是内核、文件系统、设备和应用共同形成的协议。只理解其中一层，无法判断崩溃后真实结果。

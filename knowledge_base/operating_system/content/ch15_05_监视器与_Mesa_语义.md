# 监视器与 Mesa 语义

## 监视器抽象

监视器是一种把互斥锁、共享状态和条件变量封装在一起的并发抽象。线程进入监视器时必须持有互斥锁，在监视器内部检查或修改共享状态；条件不满足时，通过条件变量等待。

这种结构让同步规则集中在同一模块中，避免调用者直接操作内部锁和等待队列。典型用途包括有界缓冲区、读写者控制、线程池任务队列和资源池。

## 条件变量

条件变量不是条件本身，而是等待某个状态变化的队列。线程在持有锁时检查谓词；若谓词不成立，调用 wait。wait 会原子地释放锁并让线程睡眠，被唤醒后重新获得锁再返回。

signal 或 broadcast 只表示“状态可能变化了”，不表示条件一定成立。真正的条件仍然由共享状态谓词决定。

## Mesa 语义

Mesa 语义下，signal 只是把等待线程变为可运行；发出 signal 的线程继续执行并持有锁，等待线程稍后重新竞争锁。等它真正恢复时，状态可能已经被其他线程改变。

因此等待条件必须用 while 循环检查，而不是 if。`while (!condition) wait(cv, lock);` 是 Mesa 条件变量的基本写法。醒来后重新检查谓词，是防止虚假唤醒、竞争改变和广播唤醒的必要步骤。

## Hoare 语义对比

Hoare 语义中，signal 会立即把监视器控制权交给等待线程，等待线程醒来时条件被保证成立。它语义更强，但实现和调度更复杂。Mesa 语义更接近多数现代系统实现，性能和实现更简单。

因为 Mesa 语义较弱，程序员必须显式写出谓词循环。把 signal 当成“条件已经成立”的通知，是并发程序中常见错误。

## 正确用法

所有对共享谓词的检查和修改都必须在同一把锁保护下完成。修改状态后，如果可能让等待者继续，应 signal 或 broadcast。等待者醒来后重新检查，并在条件满足时执行。

监视器把互斥和条件同步结合起来，但不会自动解决死锁、遗漏唤醒或错误谓词问题。正确性仍取决于清晰的不变量和锁顺序设计。

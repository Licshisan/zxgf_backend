# 预写日志 WAL 基本思想

## 先记录意图

预写日志的基本原则是：在覆盖最终位置之前，先把即将进行的更新写入一个已知的日志区域。崩溃后，系统读取日志即可知道哪些更新已经准备提交，哪些更新未完成，从而不必扫描整个文件系统。

在文件系统中，预写日志通常称为 journaling。它把一次高层文件系统操作涉及的多个块组织成事务，使这些分散更新获得近似原子提交语义。

## 日志事务

一个物理日志事务通常包含事务开始记录、若干更新块、事务提交或结束记录。开始记录描述本事务会更新哪些最终位置，数据部分保存要写入的块内容，提交记录表示事务已经完整写入日志。

只有当提交记录安全落盘后，事务才被视为可恢复。若崩溃时日志里只有部分事务，没有提交记录，恢复时可以忽略它；若有完整提交记录，就可以重做事务，把日志中的块写到最终位置。

## 写入顺序

WAL 的关键是写入顺序。必须先把日志内容写入稳定存储，再写提交记录；必须在事务提交后，才允许把对应更新安装到最终位置。否则崩溃后可能看到提交标记却缺少完整更新内容。

设备缓存和写重排会破坏这种顺序，因此文件系统需要写屏障、刷新命令或同步 I/O 保证关键顺序。日志协议本身正确，不代表底层设备一定按协议顺序持久化。

## 重做而非扫描

恢复时，日志文件系统只需检查日志区域。对完整已提交事务执行重做，把更新块复制到最终位置；对未提交事务丢弃。这样恢复时间与日志大小和未完成事务数量相关，而不是与整个文件系统容量相关。

重做操作必须幂等。即使某个更新在崩溃前已经安装到最终位置，恢复时再次写入相同内容也应安全。物理日志天然具备这一特点，因为它重写的是确定块内容。

## 成本

日志提高恢复速度和一致性，但正常运行会增加写入。数据或元数据先写日志，再写最终位置，可能造成写放大。文件系统通过批量事务、只记录元数据、异步检查点和日志循环复用降低开销。

WAL 的本质取舍是：在正常操作中多做一点受控工作，换取崩溃恢复时少做全盘搜索，并让多块更新具备清晰的提交边界。

# 多处理器调度、缓存亲和性与负载均衡

## 多处理器调度的新问题

单处理器调度只需要决定下一个在唯一 CPU 上运行的进程。多处理器系统有多个 CPU 核，可以真正并行运行多个任务，但也带来新问题：多个 CPU 访问共享调度结构需要同步；进程在不同 CPU 间迁移会影响缓存；各 CPU 负载可能不均；调度器既要利用所有核心，又要避免过多迁移和锁竞争。

多核硬件使并行成为常态，但单个进程未必能自动变快。只有多线程或多进程工作能被分散到多个 CPU 上，调度器的多处理器能力才有发挥空间。

## 缓存亲和性

进程在某个 CPU 上运行后，该 CPU 的缓存中会留下它的代码、数据和内核路径相关状态。下次仍在同一 CPU 上运行时，缓存命中率可能更高；若迁移到另一个 CPU，需要重新加载工作集，还可能触发跨核缓存一致性流量。

因此调度器倾向于保持缓存亲和性，即尽量让进程回到之前运行过的 CPU。亲和性不是绝对规则，因为负载均衡有时更重要；但无意义的频繁迁移会明显损害性能。

## 单队列多处理器调度

单队列方案把所有可运行任务放入一个全局队列，多个 CPU 从同一队列取任务。优点是概念简单，容易复用单 CPU 调度策略，也比较容易维持全局公平和负载均衡。只要队列中有任务，空闲 CPU 可以直接取走。

缺点是可扩展性差。全局队列必须加锁，CPU 数量增加后锁竞争会变重。另一个问题是缓存亲和性差：任务可能每次被不同 CPU 取走，导致缓存反复失效。调度器可以加入亲和性规则缓解，但实现会复杂起来。

## 多队列多处理器调度

多队列方案通常为每个 CPU 维护本地运行队列。每个 CPU 主要从自己的队列调度任务，队列锁竞争小，缓存亲和性好。任务留在固定 CPU 上运行，更容易复用本地缓存。

问题是负载不均。某个 CPU 的队列可能很长，另一个 CPU 却空闲；或者一个队列只剩一个任务，另一个队列有多个任务轮转。若不迁移任务，CPU 利用率和公平性都会下降。

## 负载均衡与工作窃取

解决负载不均需要迁移。空闲或负载较轻的 CPU 可以检查其他 CPU 的队列，从较重队列迁移一个或多个任务，这称为工作窃取。迁移后，空闲 CPU 得到工作，系统总体利用率提高。

工作窃取的频率需要控制。检查太频繁会增加锁竞争和跨核通信，削弱多队列的可扩展性；检查太少又会让负载不均持续存在。迁移还要考虑缓存亲和性，不能为了瞬时均衡而让任务在 CPU 间来回跳动。

## 同步与共享数据

多处理器调度器本身也是并发程序。运行队列、进程状态、优先级、CPU 空闲状态等共享数据必须用锁或其他同步方式保护。硬件缓存一致性只能保证内存读写最终可见，不能自动保证复合操作的原子性。

例如两个 CPU 同时从同一队列取任务，如果没有同步，可能取到同一个进程或破坏队列结构。调度器必须在正确性和低开销之间谨慎设计。

## 易错点

多处理器调度不是把单处理器算法简单复制 N 份。全局公平、局部缓存亲和、锁竞争和负载均衡会相互冲突。单队列简单但扩展性和亲和性差；多队列扩展性和亲和性好，但需要迁移机制处理负载不均。评价多核调度时必须同时看吞吐、延迟、迁移次数和同步开销。

# 虚拟化、并发与持久化三大主题

## 核心定义
虚拟化、并发和持久化是 OSTEP 组织操作系统知识的三条主线。虚拟化研究操作系统如何把有限的物理资源变成多个更易使用的逻辑资源；并发研究多个执行流同时运行或交错运行时如何保持正确性；持久化研究数据如何在断电、崩溃和程序退出后仍然存在并可恢复。

这三大主题不是互相孤立的章节划分，而是贯穿真实系统的三个基本问题。一个数据库进程运行时，需要 CPU 虚拟化让它和其他程序共享处理器，需要内存虚拟化管理缓冲区和地址空间，需要并发控制保护共享数据结构，也需要持久化机制保证写入数据不会因崩溃丢失。

## 虚拟化
虚拟化的目标是制造一种受控的假象。CPU 虚拟化让多个进程都像拥有自己的处理器；内存虚拟化让每个进程都像拥有连续、私有的地址空间；存储虚拟化让应用以文件和目录理解磁盘，而不是直接处理扇区和块地址。

虚拟化不是欺骗，而是通过硬件和内核协作实现的抽象。CPU 虚拟化依赖上下文切换、定时器中断和调度策略。内存虚拟化依赖地址转换、页表、TLB、保护位和页面替换。存储虚拟化依赖文件系统布局、inode、目录、缓存和块分配。虚拟化同时要求效率和隔离：抽象必须好用，但不能让开销过大，也不能让一个程序破坏另一个程序。

## 并发
并发关注的是多个活动同时存在时的正确性。它可能来自多线程程序，也可能来自多个进程、内核中断处理、设备回调或多处理器同时执行。并发的难点在于执行顺序不完全由程序文本决定，调度器可能在任意时刻切换线程，多个 CPU 可能真正同时访问共享数据。

并发错误通常具有偶发性。一个 `count++` 在源代码中看似一条语句，实际可能包含读取、加一、写回三个步骤。如果两个线程交错执行，就可能丢失更新。操作系统和并发程序因此需要锁、条件变量、信号量、原子指令、内存可见性规则等机制来约束执行顺序。

## 持久化
持久化解决的是数据长期保存问题。内存中的数据在进程退出或断电后会消失，而文件系统和存储设备提供跨时间保存数据的能力。应用看到的是文件读写，系统内部需要处理磁盘结构、块分配、缓存、日志、崩溃一致性和故障恢复。

持久化难在“写入并不等于已经安全落盘”。操作系统常使用缓存和延迟写提高性能，但这会带来崩溃窗口。如果系统在元数据更新到一半时崩溃，文件系统可能出现位图、inode、目录项不一致。FSCK、日志、写屏障和校验和等机制就是为了解决这些问题。

## 三者的联系
三大主题经常在一个机制中同时出现。例如文件系统缓存利用内存虚拟化和缓存管理提高 I/O 性能，同时要考虑多个进程并发读写文件，还要保证崩溃后元数据一致。再如内存映射文件把文件内容映射进地址空间，既涉及虚拟内存，也涉及持久存储和并发写入。

因此学习操作系统不能只按名词孤立记忆。理解一个机制时，应同时问：它虚拟化了什么资源？是否存在并发访问？状态是否需要持久保存？失败后如何恢复？这些问题能帮助判断一个设计的完整性。

## 典型例子
运行两个 `mem` 程序时，它们可能打印相同的虚拟地址，但各自维护不同的值。这体现了内存虚拟化：相同虚拟地址可以映射到不同物理位置。运行多个 CPU 密集程序时，它们看似同时运行，这是 CPU 虚拟化和调度的效果。如果多个线程共同更新同一个变量，就进入并发问题。若这些线程把结果写入文件，还必须考虑持久化语义。

## 易错点
常见错误是把并发等同于并行。并发强调多个任务在同一时间段内都在推进，即使单核上通过时间片交错执行也属于并发；并行强调多个任务在同一时刻真正运行，通常需要多个处理器。另一个错误是把持久化简单理解为“写磁盘”。真正的持久化还包括写入顺序、崩溃恢复和数据完整性。

还要注意虚拟化并不意味着资源无限。虚拟 CPU 仍受物理 CPU 数量限制，虚拟内存仍受物理内存和交换空间限制，文件系统也受磁盘容量、带宽和故障模式限制。

## 与其他知识点的关系
后续 CPU 调度、地址空间、分页、TLB、锁、条件变量、磁盘、RAID、文件系统日志和分布式文件系统，都可以分别放入这三条主线中理解。它们共同回答同一个问题：操作系统如何在有限、不可靠、会并发访问的硬件上，为应用提供简单、稳定、可控的运行环境。

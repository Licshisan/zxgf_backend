# 系统调用、用户态与内核态

## 核心定义
用户态和内核态是处理器提供的两类执行权限。用户态运行普通应用程序，权限受限；内核态运行操作系统内核，能够执行特权指令、管理页表、处理中断、访问设备并维护全局系统状态。系统调用是用户程序请求内核服务的受控入口。

应用程序不能直接操作硬件或内核数据结构。它需要通过 `open`、`read`、`write`、`fork`、`exec`、`wait`、`mmap` 等系统调用进入内核，由内核完成权限检查和实际操作。这种设计既提供服务，也建立边界。

## 为什么需要两种状态
如果所有程序都能以内核权限运行，任何程序都可以修改别的进程内存、关闭设备中断、重写磁盘或破坏文件系统。用户态/内核态的划分把普通程序限制在自己的地址空间和授权资源内，把敏感操作集中到内核中执行。

这种划分还让操作系统能实现资源复用。调度器、页表管理器、文件系统和设备驱动都需要维护跨进程共享的状态。如果应用能随意改这些状态，系统就无法保证公平、隔离和一致性。

## 系统调用流程
系统调用不是普通函数调用。普通函数调用仍在用户态执行，只改变用户栈和寄存器；系统调用需要通过特殊指令触发陷入，处理器切换到内核态，并跳转到内核预先登记的入口。内核根据系统调用号找到处理函数，读取参数，检查权限，执行服务，然后把返回值和错误码交回用户态。

以 `read(fd, buf, n)` 为例，用户程序传入文件描述符、缓冲区地址和读取长度。内核要检查文件描述符是否有效、缓冲区是否属于调用进程且可写、文件对象是否支持读取，然后从页缓存、文件系统或设备读取数据。完成后，内核返回实际读取字节数；如果失败，则返回错误并设置相应错误码。

## 用户态和内核态的边界
边界的关键是“受控进入”和“受控返回”。用户程序不能跳到任意内核地址，只能通过系统调用入口、异常或中断进入内核。内核处理完成后，也不能随意返回到非法用户地址。处理器和内核共同保存必要寄存器、切换栈，并恢复用户态执行现场。

异常和中断也会导致进入内核，但含义不同。系统调用通常是程序主动请求服务；异常可能来自非法访问、缺页、除零等同步事件；中断通常来自设备或定时器，是异步事件。三者都需要内核处理，但触发原因和处理路径不同。

## 典型例子
创建进程时，应用不会自己复制地址空间或修改进程表，而是调用 `fork`。内核为子进程创建 PCB，复制或共享必要的地址空间结构，复制文件描述符表，并安排父子进程从系统调用返回。父进程得到子进程 PID，子进程得到 0，这个返回值差异让同一段代码分成两条执行路径。

文件访问也是典型例子。应用以为自己只是在读写文件，实际每次关键操作都通过系统调用进入内核。内核检查权限、维护偏移量、处理缓存，并把设备差异隐藏在统一接口之后。

## 开销与设计权衡
系统调用有开销，因为它涉及权限切换、参数验证、可能的上下文状态保存以及内核路径执行。频繁的小系统调用会影响性能，所以系统常通过缓冲、批量 I/O、内存映射、异步 I/O 等方式减少进入内核的次数。

但不能为了减少开销而取消边界。用户态/内核态划分是系统安全和稳定的根基。现代系统的优化通常是在保持边界的前提下减少切换成本，而不是允许应用直接任意操作硬件。

## 易错点
常见误解是把系统调用等同于库函数。很多 C 标准库函数可能封装系统调用，但并非所有库函数都会进入内核。例如字符串处理函数通常只在用户态运行，而 `read`、`write` 这类接口才需要内核服务。另一个误解是认为进入内核就一定发生进程切换。系统调用只是权限切换，是否切换到另一个进程取决于该调用是否阻塞、调度器是否运行等因素。

还要注意内核态不是“更快模式”，而是“更高权限模式”。内核态代码写错的后果通常更严重，因为它可以破坏全局状态。

## 与其他知识点的关系
系统调用是进程 API、文件 API、内存管理和设备 I/O 的共同入口。受限直接执行机制解释了用户程序如何高效运行又能被内核控制；陷入和中断处理解释了权限切换细节；文件描述符、fork、exec、wait 等知识点都是系统调用边界上的具体例子。

# 远程过程调用 RPC

## RPC 的目标

RPC 试图把网络通信包装成类似本地函数调用的形式。客户端调用一个本地 stub，stub 把参数编码成消息发送给服务器；服务器 stub 解码参数，调用真正服务例程，再把返回值编码发回客户端。

这种抽象降低了编写分布式程序的门槛。程序员可以按“调用服务”的方式组织代码，而不必在每处手写套接字收发、编码和重试逻辑。

## 参数编码

RPC 必须把内存中的参数转换为可在网络上传输的字节序列，这称为序列化或编组。整数大小、字节序、字符串长度、结构体布局和指针都需要明确规则。接收端再按同一格式反序列化。

指针尤其麻烦。本地指针只在当前进程地址空间有意义，不能直接传给远程机器。RPC 接口通常只允许传递值、数组、字符串或显式对象标识符。

## 调用语义

本地调用失败方式较少，RPC 失败方式很多。请求可能没到服务器，服务器可能执行后崩溃，回复可能丢失，客户端可能超时重试。于是 RPC 很难保证真正的“恰好一次执行”。

常见语义包括最多一次、至少一次和近似恰好一次。至少一次通过重试保证请求最终可能执行，但可能重复；最多一次通过请求 ID 和重复过滤避免重复执行，但服务器崩溃后状态恢复复杂。

## 幂等操作

幂等操作多次执行结果等同一次执行，例如读取某块数据、设置属性为固定值。非幂等操作重复执行会改变结果，例如账户加款、追加日志或创建唯一对象。

RPC 协议常鼓励把请求设计成幂等，或为非幂等请求加入唯一 ID、事务记录和去重表。否则超时重试会把网络故障转化为业务错误。

## RPC 的边界

RPC 隐藏消息格式，但不能消除分布式本质。远程调用延迟更高，失败更复杂，服务器负载和权限检查也不同。把 RPC 当成本地调用使用，容易写出性能差且恢复困难的系统。

合理使用 RPC，应让接口粗粒度、语义明确、错误可处理，并明确调用是否可重试、是否幂等、是否需要认证和超时控制。

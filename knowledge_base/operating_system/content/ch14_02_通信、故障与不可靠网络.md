# 通信、故障与不可靠网络

## 网络为什么不可靠

网络数据包可能丢失、损坏、乱序、重复或长时间延迟。路由器和交换机缓冲区有限，突发流量会导致丢包；链路可能中断；接收端也可能因为缓冲区不足而丢弃数据。即使所有机器都没有崩溃，通信也可能失败。

分布式协议必须把“不收到消息”视为正常事件，而不是异常小概率。发送方通常无法直接区分消息丢失、回复丢失、服务器宕机或网络分区。

## 不可靠消息层

UDP 提供典型的不可靠数据报服务。发送方把数据报交给网络，接收方可能收到，也可能收不到；协议本身不保证重传、顺序或恰好一次交付。UDP 通常包含校验和，用于检测部分传输损坏，但不解决丢包。

不可靠层并非无用。某些应用可以自行处理丢包，例如实时音视频宁愿丢弃旧包，也不愿等待重传。端到端思想认为，可靠性需求应由真正需要它的端点实现。

## 确认与超时

构建可靠通信的基本方法是确认和超时。发送方发送请求后保留副本，接收方收到后返回确认。若发送方在超时时间内未收到确认，就重发请求。

这个机制能处理请求丢失，但也会产生重复请求。若原请求已到达，只是确认丢失，发送方重传后服务器会再次看到同一请求。协议必须识别重复，否则可能重复执行操作。

## 序列号与重复检测

序列号用于标识消息。发送方为请求编号，接收方记录已经处理过的编号。若收到重复编号，请求不应再次交给应用处理，但可以重新发送确认，让发送方停止重试。

重复检测的难点是状态管理。记录所有历史请求不现实，协议通常使用连接、会话、窗口或租约限制需要记住的范围。服务器崩溃后若丢失这些状态，恰好一次语义就会受到影响。

## 超时选择

超时太短会造成不必要重传，加重拥塞；超时太长又会让故障检测迟缓。网络延迟变化大时，固定超时很难适应。实际协议常根据往返时间估计动态调整超时。

通信可靠性不是单个机制解决的，而是校验和、确认、重传、序列号、流控、拥塞控制和故障恢复共同作用的结果。

# 调度指标：周转时间、响应时间与公平性

## 为什么需要指标

调度策略不能只凭直觉评价。不同算法可能在某些负载下表现很好，在另一些负载下很差。调度指标提供了比较标准，让我们能明确讨论“好”究竟指更快完成、更快响应、更公平分享 CPU，还是更符合优先级和实时期限。

OSTEP 在介绍调度时先使用简化工作负载，是为了看清指标和算法之间的关系。真实系统的进程会随时到达、运行时间未知、频繁 I/O、优先级不同，但基础指标仍然是分析调度策略的入口。

## 周转时间

周转时间指任务从到达到完成经历的总时间：

`周转时间 = 完成时间 - 到达时间`

它衡量系统把任务“做完”需要多久，适合批处理、后台计算、构建任务等场景。若所有任务同时到达，周转时间就等于完成时刻；若任务陆续到达，就必须扣除到达时间。

优化平均周转时间通常倾向于先运行短任务。短任务尽快完成，可以减少它们在系统中的等待时间，从而降低平均值。但这可能牺牲长任务的等待体验。

## 响应时间

响应时间指任务从到达到第一次被调度运行的时间：

`响应时间 = 首次运行时间 - 到达时间`

响应时间反映交互性。终端、图形界面、编辑器、Web 服务请求等都需要尽快给用户反馈，即使任务整体还没有完成。一个调度器如果总是让短任务完整运行完再处理其他任务，可能有不错的周转时间，却让后到或排在后面的交互任务迟迟得不到第一次运行机会。

轮转调度改善响应时间，因为它把 CPU 分成时间片，让多个就绪任务较快获得运行机会。

## 公平性

公平性关注 CPU 是否按合理比例分配。最简单的公平是每个同等级任务获得近似相同的 CPU 时间；更复杂的公平会考虑权重、优先级、用户份额和资源限制。公平性并不总与性能一致。为了最小化平均周转时间，调度器可能偏爱短任务；为了公平，调度器必须避免长任务长期饥饿。

公平还涉及可预测性。用户不一定要求每个任务最快完成，但希望后台任务不会永远不动，交互任务不会被长计算完全压制。

## 指标之间的冲突

调度设计经常面对冲突。短时间片让响应时间更好，但上下文切换更多，吞吐可能下降。SJF 类策略降低平均周转时间，但需要知道或估计运行时间，且可能让长任务等待很久。严格公平能避免饥饿，但可能降低对短任务或交互任务的优化空间。

因此评价调度器必须说明工作负载和目标。批处理系统、桌面系统、服务器、实时系统对指标的优先级不同。没有一种通用算法在所有指标上同时最优。

## 易错点

不要把周转时间和响应时间混为一谈。任务第一次运行很快，不代表很快完成；任务总完成时间短，也不代表用户很快看到反馈。公平也不是平均周转时间的同义词，它关心资源分配关系而非单个任务完成时刻。

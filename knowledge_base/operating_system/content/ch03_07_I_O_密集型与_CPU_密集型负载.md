# I/O 密集型与 CPU 密集型负载

## 两类负载的区别

CPU 密集型进程的大部分时间用于计算，很少主动阻塞。例如视频编码、科学计算、编译大型项目中的某些阶段，都可能持续消耗 CPU。I/O 密集型进程则经常等待磁盘、网络、终端、管道或用户输入。它们每次运行时间可能很短，随后就发起 I/O 并进入阻塞。

这种区别不是进程的永久标签，而是某段时间内的行为特征。同一个程序可能先读入大量数据，表现为 I/O 密集；随后进行计算，表现为 CPU 密集；最后写回结果，又变成 I/O 密集。

## 对调度的影响

I/O 密集型进程通常需要较好的响应时间。它们运行一小段就阻塞，如果被唤醒后能迅速获得 CPU，就可以继续处理输入或提交下一次 I/O，从而让用户感觉系统灵敏，也能提高设备利用率。若唤醒后长期排队，设备和用户都可能等待 CPU，整体效率下降。

CPU 密集型进程更关注长期吞吐和公平份额。它们即使晚一点获得 CPU，通常不会影响交互反馈；但如果长期被交互任务压制，也会出现饥饿。因此调度器需要既偏向短交互运行，又保证长计算任务持续前进。

## I/O 与 CPU 的重叠

合理调度可以让一个进程等待 I/O 时，另一个进程使用 CPU。这样 CPU 和设备并行工作，系统吞吐更高。假设进程 A 发起磁盘读后阻塞，调度器立即运行进程 B；磁盘完成后中断唤醒 A，A 再进入就绪队列。这个过程比让 CPU 空等磁盘完成有效得多。

这也是多道程序和分时系统的重要收益。只要系统中有足够的可运行任务，单个进程的等待时间就不必转化为 CPU 空闲时间。

## 调度器如何识别

操作系统通常不知道进程未来的完整行为，只能根据历史观察推断。频繁在时间片用完前阻塞的进程，往往被认为更偏交互或 I/O 密集；长时间用完整个时间片的进程，则更像 CPU 密集。MLFQ 正是利用这种反馈调整优先级。

这种推断可能出错。某些程序可能刻意在时间片末尾前发起无意义 I/O 来维持高优先级；某些计算程序也可能阶段性等待 I/O。调度器需要累计用量、周期性提升优先级和限制作弊，避免单一规则被利用。

## 性能取舍

偏向 I/O 密集型任务可以改善交互响应和设备流水，但如果偏向过强，CPU 密集型任务会进展缓慢。偏向 CPU 密集型任务可以减少切换开销，提高长任务吞吐，却可能让终端、编辑器、网络服务变得迟钝。

实际系统常结合优先级、时间片长度、睡眠唤醒奖励、CPU 使用衰减和用户 nice 值来平衡。高优先级短时间片适合交互任务；低优先级长时间片适合批处理任务。

## 易错点

I/O 密集不等于“不需要 CPU”。I/O 完成后，它仍需 CPU 处理数据、提交下一次请求或响应用户。CPU 密集也不等于“不重要”，后台计算、服务端批处理和系统维护任务都需要公平进展。调度分析应看任务的运行和阻塞交替模式，而不是只看程序名称。

# 定时器中断与上下文切换

## 定时器中断的作用

定时器中断是操作系统抢占 CPU 的基础。内核启动时配置硬件定时器，使其周期性地产生中断。即使用户进程不发起系统调用，定时器也会把 CPU 控制权交回内核。内核据此统计运行时间、处理时间片到期、唤醒睡眠任务，并决定是否进行调度。

没有定时器中断，系统只能依靠进程主动让出 CPU。这样的协作式模型无法处理无限循环、恶意程序或计算密集型任务长期不进入内核的问题。

## 上下文切换是什么

上下文切换是内核暂停当前执行流并恢复另一个执行流的过程。对进程而言，它通常包括保存当前进程寄存器、切换内核栈和调度状态、切换地址空间、恢复目标进程寄存器，并返回到目标进程的执行位置。切换后，CPU 继续执行的已经是另一个进程。

上下文切换让每个进程都感觉自己能连续使用 CPU，实际则是多个进程按时间片交替运行。它是 CPU 虚拟化的核心机制之一。

## 切换流程

当定时器中断到来时，硬件先保存必要现场并进入内核中断处理程序。内核更新当前进程的 CPU 使用统计，判断它是否需要继续运行。如果时间片未耗尽且没有更高优先级任务，内核可以直接返回当前进程。若需要调度，调度器选择下一个可运行进程。

随后内核保存当前进程的完整上下文，把它放回就绪队列或其他状态队列；再从目标进程 PCB 中取出上下文，恢复页表、内核栈和寄存器。最后执行返回用户态的指令，CPU 从目标进程上次停止的位置继续。

## 成本来源

上下文切换不是免费操作。直接成本包括保存和恢复寄存器、进入和退出内核、队列操作以及调度算法本身。间接成本更隐蔽：切换进程可能导致 CPU 缓存、TLB 和分支预测状态失效，新进程刚运行时需要重新装入工作集。

因此时间片不能无限缩短。短时间片改善响应时间，但增加切换频率；长时间片摊薄切换成本，却可能让交互任务等待太久。调度器必须在延迟和吞吐之间折中。

## 与系统调用切换的区别

系统调用也会从用户态进入内核，但不一定发生上下文切换。进程调用 `read()` 如果数据立即可用，内核可以完成请求后返回同一进程。只有当进程阻塞、时间片到期、优先级变化或内核主动重新调度时，才需要切换到另一个进程。

同样，中断也不必然导致切换。定时器中断提供调度机会，但是否切换取决于调度策略和当前系统状态。

## 易错点

上下文切换不是应用层函数切换，也不是线程库中的普通回调。它涉及特权级、硬件现场、内核数据结构和调度决策。另一个误区是把“中断发生”理解为“当前进程已经结束”。实际上大多数中断处理后，当前进程可能继续运行。

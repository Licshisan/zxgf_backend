# 非死锁并发缺陷与事件驱动并发

## 非死锁缺陷

并发错误不只有死锁。实际大型软件中，更多问题来自违反原子性和违反顺序。违反原子性是代码期望一组访问不可分割，但实际被其他线程插入；违反顺序是某操作必须在另一操作之后执行，但缺少同步导致顺序被打破。

例如一个线程检查指针非空后使用，另一个线程在中间把指针置空，这就是原子性被破坏。一个线程使用对象前假设初始化已完成，但初始化线程尚未运行，就是顺序错误。

## 修复方式

违反原子性通常用锁修复，把检查和使用放在同一临界区内。违反顺序通常用条件变量、信号量、join 或其他事件通知修复，明确建立先后关系。关键是先说明程序依赖的不变量或顺序，再选择同步原语。

只靠 sleep 或调度经验不能修复并发缺陷，它们只改变概率。

## 事件驱动并发

事件驱动并发用单线程事件循环处理多个 I/O 事件。程序等待事件集合，某个描述符可读、可写或定时器到期后，调用对应处理程序。因为同一时刻只有一个处理程序运行，基本事件循环中不需要锁来保护共享状态。

网络服务器、GUI 和 JavaScript 运行时常使用这种模型。它把调度控制交给程序员，避免多线程交错带来的部分问题。

## 事件模型的限制

事件处理程序不能阻塞。如果在事件循环中执行阻塞磁盘 I/O、长计算或等待锁，整个系统都无法处理其他事件。解决方法包括异步 I/O、非阻塞接口、线程池处理阻塞任务，或把长任务拆成多个小事件。

事件驱动也不自动简单。状态会被拆散到多个回调中，错误处理和控制流可能变得复杂。

## 易错点

事件驱动减少了锁竞争，但不消除所有并发问题。异步回调之间仍可能有顺序错误，多个事件共享状态也需要清晰状态机。选择线程还是事件循环，应看任务是否 I/O 密集、是否需要多核并行、阻塞操作能否异步化。

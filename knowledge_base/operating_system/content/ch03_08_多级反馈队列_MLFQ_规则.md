# 多级反馈队列 MLFQ 规则

## MLFQ 要解决的问题

多级反馈队列试图同时获得两类好处：像 SJF/STCF 那样让短任务尽快完成，又像轮转调度那样给交互任务较短响应时间。困难在于操作系统通常不知道任务还要运行多久，只能在运行过程中观察行为并调整判断。

MLFQ 的基本思想是建立多个优先级队列。高优先级队列先于低优先级队列运行；同一队列内部使用轮转。进程的优先级不是固定不变，而是根据它过去如何使用 CPU 动态变化。

## 基本规则

第一条规则：如果进程 A 的优先级高于进程 B，调度器运行 A。第二条规则：如果二者优先级相同，按轮转方式分享 CPU。这样，队列级别决定跨优先级选择，时间片决定同级公平。

第三条规则：新进入系统的任务放入最高优先级队列。这相当于先乐观地假设新任务可能很短或具有交互性。若它确实很快完成，就获得类似短任务优先的效果；若它持续占用 CPU，会在之后被降级。

## 根据行为调整优先级

早期简单规则是：如果任务用完整个时间片，就降低一级；如果在时间片内主动阻塞或让出 CPU，就保持当前优先级。这使 I/O 密集型或交互型任务维持高优先级，因为它们经常等待输入、磁盘或网络，不会长时间占用 CPU。

CPU 密集型任务则会逐渐下沉到低优先级队列。低队列通常使用较长时间片，以减少上下文切换成本，让长任务稳定推进。这种设计用历史行为近似预测未来：过去总是短运行并阻塞的任务，未来也可能需要快速响应；过去持续用满 CPU 的任务，未来也可能是长计算。

## 时间片与队列层次

不同层级可以配置不同时间片。高优先级队列使用短时间片，保证交互任务快速轮换；低优先级队列使用长时间片，减少长任务频繁切换造成的开销。队列层数、每层时间片和降级规则都是策略参数。

这些参数没有放之四海皆准的最佳值。系统负载、用户期望、硬件速度和应用类型都会影响选择。真实系统往往使用经验默认值，并允许管理员或用户通过配置、优先级建议或 nice 值进行微调。

## MLFQ 的效果

短任务进入最高优先级后可能在降级前完成，从而获得较低周转时间。交互任务频繁阻塞并被唤醒，通常保持较高优先级，响应时间较好。长 CPU 任务最终进入低队列，仍能在没有高优先级竞争或被周期提升时运行。

因此 MLFQ 不是某个单一指标的最优算法，而是一种综合折中。它把“调度器不知道任务长度”的问题转化为“根据历史行为不断修正优先级”的问题。

## 易错点

MLFQ 的“反馈”指根据进程实际运行行为调整优先级，不是用户手动指定多个队列。高优先级也不表示进程永远重要，它可能只是刚进入系统或近期表现为交互型。理解 MLFQ 时应同时关注入队、降级、同级轮转、时间配额和周期提升。

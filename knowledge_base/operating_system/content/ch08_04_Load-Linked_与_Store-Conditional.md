# Load-Linked 与 Store-Conditional

## LL/SC 的基本语义

Load-Linked 和 Store-Conditional 是另一类实现原子更新的硬件原语。LL 从某地址读取值，并在处理器中记录这个地址的监视状态。SC 尝试写入该地址；如果自 LL 以来没有其他处理器修改过该地址，SC 成功并写入；否则失败。

LL/SC 表达的是“读取后，如果期间没人改过，就提交写入”。它可以实现 CAS 类效果，也能构建锁和无锁结构。

## 与 CAS 的区别

CAS 比较的是内存当前值是否等于期望值；LL/SC 检测的是从 LL 到 SC 之间是否发生过干扰写。由于关注修改事件，LL/SC 在某些情况下可避免 ABA 问题：即使值变回原值，只要中间被写过，SC 也可能失败。

但 LL/SC 对实现条件敏感。上下文切换、异常、中断或访问其他地址可能清除链接状态，导致 SC 失败。因此代码通常在短循环中使用 LL/SC。

## 实现自旋锁

用 LL/SC 实现锁时，线程先 LL 读取锁值。如果锁为 1，说明已占用，继续等待；如果为 0，就用 SC 尝试写入 1。SC 成功则获得锁，失败说明有竞争，重新尝试。释放锁时普通写 0 即可，但要配合合适内存屏障。

这种实现避免了把“读取”和“写入”拆成可竞争窗口。

## 使用约束

LL/SC 循环应尽量短，避免在 LL 和 SC 之间执行复杂逻辑、函数调用或可能异常的访问。否则 SC 失败率高，性能下降。硬件通常只保证在有限条件下 SC 能成功，不能把它当作任意长事务。

多处理器上，LL/SC 仍会产生缓存一致性流量，但可以减少无条件原子写。

## 易错点

SC 失败不一定表示有逻辑错误，它可能因为竞争或硬件清除链接状态而失败。LL/SC 不是数据库事务，不能保护任意多内存位置的复杂更新。它只是一对低级原子原语，需要同步算法正确使用。

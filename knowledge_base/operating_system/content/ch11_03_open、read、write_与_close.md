# open、read、write 与 close

## open

`open` 把路径名转换为一个可访问的文件描述符。调用时可以指定访问模式，如只读、只写、读写，也可以指定创建、截断、追加、同步等标志。带 `O_CREAT` 时，内核还需要根据权限位和 umask 创建目录项与 inode。

`open` 的工作包括路径解析、权限检查、分配打开文件对象、初始化偏移量和状态标志、在进程描述符表中安装表项。返回值只是一个整数，但它背后代表一组已经建立好的内核状态。

## read

`read(fd, buf, size)` 从描述符对应对象读取最多 `size` 字节到用户缓冲区。普通文件读取会从当前偏移开始，成功后偏移增加实际读取字节数。返回 0 通常表示已经到达文件末尾，返回负值表示错误。

实际读取字节数可能小于请求大小。普通文件可能因为到达 EOF 而少读；管道、终端、套接字等对象更常见短读。健壮程序不能假定一次 `read` 一定填满缓冲区，而应根据返回值循环处理。

## write

`write(fd, buf, size)` 把用户缓冲区中的字节写入文件或其他对象。普通文件写入会从当前偏移开始，成功后偏移前移。若以追加模式打开，系统会在每次写入前把位置设为文件末尾，从而避免多个追加写互相覆盖。

`write` 返回成功通常只表示数据已被内核接收，不一定已经写到磁盘。错误也可能延迟暴露，例如后台回写失败可能在之后的 `fsync` 或 `close` 时才被报告。对可靠存储有要求的程序不能只依赖 `write` 返回值。

## close

`close` 释放进程描述符表中的表项，并减少打开文件对象引用计数。当引用计数降为零时，内核可释放打开文件对象；若底层文件也没有目录引用，文件数据可能被回收。`close` 还可能触发缓冲区刷新或报告此前延迟的错误。

关闭描述符后，该整数可被之后的 `open` 重用。程序若继续使用已关闭描述符，可能误操作另一个新打开文件，这是常见资源管理错误。多线程程序中尤其要谨慎，因为一个线程关闭描述符，另一个线程可能仍在使用同一整数。

## 调用顺序与错误处理

典型文件访问流程是 `open`、循环 `read` 或 `write`、必要时 `fsync`、最后 `close`。每一步都应检查返回值。`open` 失败可能是路径不存在或权限不足；`read` 和 `write` 可能被信号中断；`close` 也可能失败并携带写回错误。

这些接口看似简单，但它们连接了路径命名、权限、缓存、偏移量、持久化和资源释放。理解每个调用改变了哪些内核状态，比单纯记函数原型更重要。

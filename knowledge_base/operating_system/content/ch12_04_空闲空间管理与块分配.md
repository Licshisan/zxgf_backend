# 空闲空间管理与块分配

## 为什么空闲空间管理重要

文件系统必须持续分配和释放 inode 与数据块。若空闲空间管理粗糙，文件会被分散到磁盘各处，顺序访问变成随机访问；若元数据更新不严谨，崩溃后可能出现重复分配或空间泄漏。

空闲空间管理同时影响性能和正确性。它既要快速找到可用空间，又要尽量提供连续块，还要在删除和崩溃恢复后保持分配状态可信。

## 位图方式

位图是常见分配结构。每个 inode 或数据块对应一位，文件系统扫描位图寻找 0 位并置为 1。位图紧凑，适合判断一段连续块是否空闲，也容易统计空闲空间。

位图的缺点是扫描大位图可能有开销，因此真实系统会维护空闲计数、分组摘要或多级结构，加速定位。即便如此，位图仍是理解文件系统分配的基础。

## 空闲列表方式

空闲列表把空闲块串成链表或分组链。分配时从链表取块，释放时放回链表。这种方式实现简单，但难以快速找到连续空间，也不容易观察全局布局。

早期文件系统若只按空闲列表顺序分配，经过大量创建删除后，逻辑连续文件可能占用相距很远的块，导致碎片严重。磁盘碎片整理工具本质上就是重新排列数据和元数据，恢复较好的局部性。

## 块分配策略

好的分配策略会尽量把同一文件的数据块连续放置，把文件 inode 和数据放近，把同一目录下相关文件放在相近区域。这样可以减少寻道，提高顺序读写吞吐。

但连续分配并不总能做到。文件增长不可预测，空闲空间会被历史操作切碎，大文件也可能占满局部区域。文件系统需要在局部连续、全局均衡和未来扩展空间之间折中。

## 元数据更新风险

分配一个块通常要更新数据位图和 inode 指针，还可能写入数据块本身。如果崩溃发生在这些更新之间，可能出现位图显示已分配但 inode 未引用，或 inode 引用但位图显示空闲。

前者造成空间泄漏，后者可能导致同一块被分配给两个文件。文件系统检查、日志和写时复制等机制都是为了解决这类多结构更新的原子性问题。

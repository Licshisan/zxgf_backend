# 锁粒度与性能权衡

## 粗粒度锁

粗粒度锁用一把锁保护较大范围的数据或操作。例如整个哈希表一把锁，整个文件系统缓存一把锁。优点是简单，容易保证不变量，死锁风险较低，代码维护成本小。

缺点是并发度低。即使两个线程访问无关对象，也会因为同一把锁互相等待。随着 CPU 数增加，粗粒度锁可能成为扩展性瓶颈。

## 细粒度锁

细粒度锁把保护范围拆小，例如每个链表节点一把锁、每个哈希桶一把锁、每个 CPU 一个局部计数器锁。这样无关操作可以并行，提高吞吐。

代价是复杂性上升。代码需要清楚锁顺序、锁覆盖范围和不变量边界。多个锁组合时容易死锁，遗漏某个路径也更难发现。

## 锁拆分与分片

锁拆分把原本由一把锁保护的数据拆成多个独立锁。锁分片则把大量对象映射到有限数量的锁上，例如哈希桶锁。二者都能降低竞争，但不会消除共享热点。

如果所有线程仍集中访问同一个桶或同一个全局计数，细粒度设计也可能退化。性能优化要基于实际竞争位置，而不是机械增加锁数量。

## 局部计数器案例

全局计数器一把锁在多核上扩展性差。懒惰计数器为每个 CPU 维护局部计数，局部更新只争用本地锁，达到阈值后再合并到全局计数。阈值越大，性能越好，但全局读数越不精确。

这个例子体现常见取舍：准确性、实时性和扩展性不能总是同时最优。

## 评估方法

选择锁粒度应看临界区长度、竞争频率、CPU 数、读写比例和不变量复杂度。无竞争代码中，细粒度可能只是增加开销；高竞争场景中，粗粒度会严重限制并行。

基准测试应覆盖实际负载。只测单线程或只测理想均匀访问，可能掩盖热点和锁顺序问题。

## 易错点

细粒度锁不是自动高级方案。若设计不清，它会让程序更慢、更难维护、更容易死锁。先用简单正确方案，再根据测量结果优化，通常比预先设计复杂锁层级更稳妥。

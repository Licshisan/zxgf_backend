# 进程间通信 IPC 基础

## 为什么需要 IPC

进程之间默认相互隔离。每个进程拥有自己的地址空间，一个进程不能直接读写另一个进程的普通内存。这种隔离提高了安全性和可靠性，但应用又经常需要协作：shell 管道要传递数据，服务器和工作进程要分派任务，图形程序要与窗口系统通信，父进程要获知子进程状态。进程间通信即 IPC，用来在隔离边界上提供受控的数据交换和事件通知。

IPC 的设计目标通常包括正确传递数据、控制访问权限、支持同步、降低复制开销以及在异常退出时保持资源可回收。不同 IPC 机制在这些目标之间取舍不同。

## 管道与命名管道

管道是最经典的 IPC 机制之一，提供单向字节流。匿名管道通常由有亲缘关系的进程使用，因为文件描述符需要通过 `fork()` 继承。它适合流式数据处理，例如一个命令产生文本，另一个命令逐行过滤。

命名管道也称 FIFO，在文件系统中有名字，无亲缘关系的进程也可以打开同一个 FIFO 通信。管道的优点是接口简单，能自然配合阻塞读写和标准输入输出；缺点是它是字节流，不保留应用层消息边界。若要传递结构化消息，应用需要自己定义分隔、长度或编码协议。

## 消息、套接字与信号

消息队列以消息为单位传递数据，接收方可以按消息读取，边界比管道清晰。套接字既可用于同一台机器上的本地通信，也可用于网络通信，是客户端和服务器模型的基础。Unix domain socket 还可以在本机进程间传递文件描述符，能力比普通管道更强。

信号更偏向事件通知而不是大量数据传输。它可以告诉进程发生了某个异步事件，例如终止请求、子进程退出或定时器到期。信号处理需要谨慎，因为它可能打断正常执行流，处理函数中能安全调用的操作有限。

## 共享内存

共享内存允许多个进程把同一段物理内存映射到各自地址空间中，从而避免频繁复制数据。它适合大数据量、高频通信的场景，例如数据库、浏览器多进程架构或高性能计算。

共享内存的难点在同步。因为多个进程能同时访问同一数据结构，必须使用信号量、互斥锁、条件变量、原子操作或其他同步机制保护一致性。没有同步的共享内存会带来与多线程类似的数据竞争，而且调试更困难，因为参与者是多个独立进程。

## IPC 与文件描述符模型

UNIX 把许多 IPC 对象也纳入文件描述符模型。管道、套接字、终端和普通文件都可以通过 `read()`、`write()`、`close()` 等接口操作。这种统一接口让 shell 组合命令变得简单，也让事件循环可以用 `select`、`poll`、`epoll` 等机制同时等待多个 I/O 来源。

文件描述符继承和传递也是 IPC 的重要部分。父进程可以在 `fork()` 后让子进程继承管道端；本地套接字还可以把已经打开的描述符发送给另一个进程。这样进程之间传递的不只是字节，也可以是访问某个内核对象的能力。

## 选择 IPC 机制

选择 IPC 时应看通信模式和数据规模。简单父子单向流适合管道；本机客户端服务器适合 Unix domain socket；跨机器通信需要网络套接字；大块共享数据适合共享内存加同步；只通知事件可使用信号或事件文件描述符。还要考虑权限、生命周期、阻塞行为和异常清理。

一个常见原则是先选语义清晰的机制，再优化性能。共享内存虽然快，但需要额外处理同步和崩溃恢复；管道虽然会复制数据，却简单可靠。机制越底层，应用承担的不变量越多。

## 易错点

IPC 不是取消进程隔离，而是在内核控制下建立通信通道。管道和套接字传输的是字节或消息，共享内存才允许直接共享数据区域。阻塞读写不是错误，而是同步语义的一部分；但如果描述符未关闭、协议边界不清或双方等待顺序设计错误，就可能出现死锁、卡住或数据解析错误。

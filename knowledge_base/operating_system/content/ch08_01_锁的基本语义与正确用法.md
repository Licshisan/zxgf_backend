# 锁的基本语义与正确用法

## 锁解决什么问题

锁用于保护临界区，让一段访问共享可变状态的代码在同一时刻只被一个线程执行。没有锁时，多个线程的读改写操作可能交错，导致丢失更新或破坏数据结构。加锁后，线程进入临界区前必须先获得锁，离开时释放锁。

锁提供的是互斥，不是自动修复逻辑错误。它要求所有访问同一共享状态的路径都遵守同一套加锁规则。

## lock 与 unlock

`lock()` 的语义是尝试获得锁。若锁空闲，调用线程成为持有者并继续执行；若锁已被其他线程持有，调用线程等待。`unlock()` 释放锁，使其他等待线程有机会进入临界区。

锁变量本身保存锁状态，也可能包含等待队列、持有者信息、调试标志等。使用者通常不直接操作这些内部字段，而通过库函数访问。

## 正确使用方式

进入临界区前加锁，退出临界区后解锁。临界区应覆盖共享状态不变量被破坏到恢复的全过程。例如链表插入时，从修改新节点指针到更新头指针都应在同一锁保护下。

错误路径也必须释放锁。实际代码常使用统一退出路径、作用域锁或 `try/finally` 风格结构，避免异常返回、错误分支或提前 return 时忘记解锁。

## 锁保护对象

一把锁应该有明确保护对象。可以是一整个数据结构，也可以是其中某个字段或桶。文档和命名应让维护者知道访问某状态前需要哪把锁。

若同一状态被多把锁“有时保护”，就容易出现遗漏和死锁。若无关状态共用一把大锁，正确性简单但并发度低。

## 评价锁

锁实现可从互斥、公平和性能评价。互斥是基本要求；公平关注等待线程是否可能长期饥饿；性能关注无竞争加锁成本、竞争时等待成本、多处理器扩展性和缓存影响。

不同锁适合不同场景。短临界区在多核上可用自旋锁；长等待或可能阻塞的临界区更适合休眠锁。

## 易错点

锁不是越多越安全。过多锁会增加死锁和维护复杂度。持锁期间执行阻塞 I/O、调用未知回调或获取其他锁都要谨慎。锁只保护按约定使用它的代码，无法阻止未加锁路径破坏共享状态。

# 共享数据与竞态条件

## 共享数据的问题

多个线程共享地址空间，因此可以同时访问同一变量、数组、链表或对象。只读共享通常没有问题；一旦至少一个线程会写入，就需要考虑并发控制。没有同步的读写可能使结果依赖调度时序。

共享数据的风险不在于“多个线程存在”，而在于多个线程对同一状态执行非原子复合操作。例如 `counter = counter + 1` 看似一行代码，实际包含读取、加一、写回多个步骤。

## 竞态条件

竞态条件指程序结果依赖不可控执行顺序。两个线程同时执行自增，可能都读到旧值 5，然后分别写回 6，最终只增加一次。正确结果应为 7，但某种交错导致更新丢失。

竞态条件常表现为偶发错误。运行一次正确不代表没有问题，只是调度交错刚好没有触发错误。CPU 核数、编译优化、负载、I/O 时序都会改变交错。

## 数据竞争

数据竞争通常指两个线程在没有同步的情况下访问同一内存位置，且至少一个是写操作。数据竞争会破坏语言和硬件内存模型的假设，在 C/C++ 中甚至可能导致未定义行为。

竞态条件是更广义的逻辑错误，数据竞争是其中常见且危险的一类。有些竞态即使每次访问都是原子的，也可能因检查和使用之间状态变化而发生。

## 不变量被破坏

共享数据结构通常有不变量。例如链表节点连接必须一致，队列头尾关系必须匹配，哈希表桶和计数要同步更新。线程在更新中间被切换，另一个线程看到半更新状态，就可能读到错误结构。

因此保护的不只是单个变量，而是整个临界区内的一组状态变更。锁的范围应覆盖不变量从被破坏到恢复的全过程。

## 识别共享状态

分析并发程序时，先列出哪些数据会被多个线程访问，再标记哪些会写入。全局变量、堆对象、静态缓存、文件描述符、日志缓冲区、引用计数和队列都是常见共享状态。函数局部变量若只在当前线程栈上，通常不是共享；但它指向的堆对象可能共享。

## 易错点

`volatile` 不能解决竞态条件。它不提供互斥，也不保证复合操作原子性。单核机器也会有竞态，因为线程可在指令之间被抢占。只有明确同步才能让共享数据更新具备可预期顺序。

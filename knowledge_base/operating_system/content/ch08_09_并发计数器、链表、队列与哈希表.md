# 并发计数器、链表、队列与哈希表

## 并发数据结构的目标

并发数据结构要同时满足正确性和性能。正确性要求操作在多线程下保持数据结构不变量；性能要求多个线程尽可能并行，避免所有操作都串行等待。最简单方案是在每个方法入口加一把大锁，通常正确但扩展性有限。

优化的核心是识别哪些操作真正冲突，哪些可以并发执行。

## 并发计数器

普通计数器用一把锁保护 `value`，`increment`、`decrement` 和 `get` 都加锁。这很简单，但所有更新竞争同一锁。多核频繁计数时，锁和缓存行会成为瓶颈。

懒惰计数器通过多个局部计数器减少竞争。每个 CPU 或线程更新本地计数，达到阈值后合并到全局。它牺牲 `get` 的实时精确性，换取更新扩展性。

## 并发链表

链表可以用整表锁保护，插入、删除、查找都串行。细化方案可以使用手递手锁：遍历时持有当前节点锁，再获取下一个节点锁，释放前一个节点锁。这样不同线程可在链表不同区域并行。

但手递手锁实现复杂，锁操作频繁。对短链表或低竞争场景，整表锁可能更快。删除节点还要小心内存回收，避免其他线程仍持有指针。

## 并发队列

队列常有入队和出队两个端。一个简单优化是分别用头锁和尾锁保护出队和入队，使生产者和消费者在多数情况下可以并行。若队列从空到非空或从一项到空，头尾状态仍需协调。

队列还常配合条件变量：消费者在队列为空时等待，生产者入队后通知。

## 并发哈希表

哈希表天然适合分片。每个桶一把锁，不同桶上的操作可以并行。查找、插入、删除只锁相关桶，竞争显著低于全表锁。若需要扩容，问题会复杂得多，因为扩容涉及全表结构变化，常需要全局锁或渐进迁移。

桶锁数量、哈希函数质量和负载因子都会影响并发性能。热点 key 会让单个桶成为瓶颈。

## 易错点

给数据结构加锁不是把每一行都包起来，而是保护结构不变量。细粒度锁需要明确锁顺序和对象生命周期。并发数据结构的性能必须测量，复杂方案在低竞争下可能不如简单大锁。

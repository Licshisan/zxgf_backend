# MLFQ 的饥饿、防作弊与优先级提升

## 基本 MLFQ 的问题

基本 MLFQ 会奖励频繁让出 CPU 的任务，惩罚持续用满时间片的任务。这有利于交互响应，但也带来三个问题：低优先级 CPU 密集型任务可能长期得不到运行；进程可能故意在时间片快用完前发起 I/O 来维持高优先级；进程行为会随时间变化，过去是 CPU 密集不代表以后仍然如此。

如果系统中一直有大量高优先级交互任务，低优先级队列可能几乎没有机会被调度。这就是饥饿。饥饿不是单次等待时间长，而是缺乏长期进展保证。

## 周期性优先级提升

解决饥饿的常见方法是周期性优先级提升。经过固定时间 S 后，系统把所有任务重新放回最高优先级队列。这样低优先级任务至少会定期获得运行机会，不会永久沉底。

优先级提升还解决行为变化问题。一个曾经长时间计算的进程，后来可能进入交互阶段；如果它永远留在低优先级，就无法获得及时响应。周期提升让调度器有机会重新观察它的近期行为。

## S 值的取舍

提升周期 S 太长，低优先级任务仍可能等待很久，交互行为变化也难以及时反映。S 太短，CPU 密集型任务频繁回到高优先级，会削弱 MLFQ 对交互任务的偏向，增加短任务等待。

这类参数常被称为经验常量。系统设计者只能结合负载测试和使用场景设定默认值，再通过配置或自适应策略调整。它体现了操作系统策略设计中常见的现实问题：机制清楚，参数却很难一劳永逸。

## 防作弊：累计时间配额

原始规则中，进程只要在时间片用完前主动阻塞，就可以保持高优先级。恶意或投机程序可以每次运行到时间片末尾前做一次无意义 I/O，从而长期占据高优先级。

改进方法是按队列层级累计 CPU 使用量，而不是每次调度重新计时。只要进程在某一层消耗完分配给它的总配额，无论中间主动让出多少次 CPU，都要降到下一层。这样交互任务仍能因短运行获得响应，但无法通过拆分运行时间无限逃避降级。

## 更稳健的反馈

真实调度器还可能使用 CPU 使用衰减、睡眠时间奖励、优先级上限、用户 nice 值和调度类来控制行为。CPU 使用衰减表示历史越久影响越小，近期行为更重要。睡眠奖励能改善交互唤醒延迟，但必须限制幅度，避免被滥用。

防作弊和优先级提升是一组互补机制：前者阻止任务不公平地留在高层，后者保证任务不会永久留在低层。

## 易错点

饥饿不是 MLFQ 必然失败，而是缺少提升和公平保护时的风险。防作弊也不是禁止 I/O 密集任务保持高优先级，而是防止任务通过人为拆分 CPU 使用量获得超额份额。评价 MLFQ 不能只背规则，还要看这些规则在长期运行中的边界行为。

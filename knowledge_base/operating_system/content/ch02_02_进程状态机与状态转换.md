# 进程状态机与状态转换

## 为什么需要状态机

操作系统同时管理大量进程，但物理 CPU 数量有限。为了描述每个进程当前能否运行、正在等待什么、是否已经结束，内核必须为进程设置状态，并在事件发生时进行转换。状态机的价值在于把复杂的执行过程拆成少数清晰阶段，使调度器、I/O 子系统和进程管理逻辑能够协同工作。

最基础的进程状态包括运行、就绪和阻塞。运行表示进程正在某个 CPU 上执行指令；就绪表示进程已经具备运行条件，但暂时没有被调度；阻塞表示进程正在等待某个外部事件，例如磁盘 I/O 完成、管道中出现数据、锁或条件变量被唤醒。

## 三态模型

在三态模型中，就绪进程进入就绪队列，等待调度器选择。调度器把某个就绪进程切换到 CPU 上后，它从就绪变为运行。运行进程可能因为时间片用完、被更高优先级进程抢占或主动让出 CPU 而回到就绪；也可能因为发起 I/O、等待子进程、等待同步事件而进入阻塞。

阻塞状态的关键是“当前即使给它 CPU 也无法继续完成有效工作”。例如进程调用 `read()` 从空管道读取数据，如果没有写端提供内容，它继续运行只会反复检查条件，浪费 CPU。正确做法是让它睡眠，等管道写入事件发生后再把它唤醒到就绪状态。

## 常见扩展状态

真实系统通常比三态模型更细。新建状态表示进程结构已经创建但尚未进入调度；终止状态表示进程正在释放资源；僵尸状态表示进程已经退出，地址空间等大部分资源已释放，但退出码、资源使用统计和 PID 仍保留，等待父进程 `wait()` 回收。

有些系统还区分可中断睡眠和不可中断睡眠。可中断睡眠可以被信号打断，常见于等待用户可感知的事件；不可中断睡眠通常用于关键内核 I/O 路径，随意打断可能破坏设备协议或内核不变量。调试系统卡顿时，这些细分状态比简单的“阻塞”更有诊断价值。

## 状态转换的触发

状态转换通常由四类事件触发。第一类是调度事件，例如时间片耗尽导致运行转就绪。第二类是系统调用，例如 `read()` 没有数据时运行转阻塞，`exit()` 导致运行转终止。第三类是硬件或设备事件，例如磁盘中断通知 I/O 完成，内核把等待该 I/O 的进程从阻塞转就绪。第四类是父子进程事件，例如子进程退出后唤醒正在 `wait()` 的父进程。

每次转换都必须同步更新内核数据结构。一个进程不能同时存在于多个不兼容队列中：如果它在磁盘等待队列里，就不应同时被普通调度器当作可运行进程；如果它已经退出，就不应再被调度到 CPU 上。

## 调度视角下的状态

调度器主要关心就绪和运行。它不直接“优化”阻塞进程，因为阻塞进程缺少继续执行的条件。合理的系统会利用阻塞提高 CPU 利用率：当一个进程等待 I/O 时，另一个就绪进程可以运行。这样即使单个进程有大量等待时间，系统整体仍能保持吞吐。

但状态转换也有成本。频繁阻塞和唤醒会带来上下文切换、队列操作和缓存失效；过长时间片会损害交互响应；过短时间片又会增加调度开销。状态机描述的是正确性边界，调度策略则决定在这些边界内如何取舍性能。

## 易错点

就绪不是“马上运行”，而是“可以运行”。阻塞不是“运行慢”，而是缺少继续执行所需事件。僵尸进程不是仍在执行的进程，而是退出信息尚未被回收的残留内核记录。理解这些区别，才能正确分析 `ps` 输出、系统负载和进程生命周期。

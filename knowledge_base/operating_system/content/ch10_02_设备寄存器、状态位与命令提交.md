# 设备寄存器、状态位与命令提交

## 设备寄存器的作用

操作系统控制设备的最直接方式是读写设备寄存器。一个简化设备通常至少包含状态寄存器、命令寄存器和数据寄存器。状态寄存器告诉内核设备是否忙、是否就绪、是否出错；命令寄存器用于启动读、写、复位等操作；数据寄存器或数据端口用于传入传出少量数据，较大的数据通常交给 DMA 机制搬运。

寄存器不是普通变量。它们连接到硬件，读写会产生副作用：读状态可能清除某些标志，写命令可能立即启动设备，写错顺序可能触发未定义行为。因此驱动程序必须按照设备手册规定的时序访问寄存器。

## 端口 I/O 与内存映射 I/O

主机与设备通信常见两种方式。端口 I/O 使用专门指令访问设备端口，例如某些体系结构上的输入输出指令。内存映射 I/O 则把设备寄存器映射到物理地址空间，内核通过普通 load/store 指令访问这些地址，但硬件把访问转发给设备而不是内存。

两者都必须受权限保护。用户程序不能随意写设备寄存器，否则可以绕过文件权限、破坏磁盘内容、读取别的进程数据，甚至让设备进入错误状态。只有内核或被授权的驱动代码可以直接操作这些接口。

## 命令提交的基本协议

一个典型提交过程包括等待设备空闲、准备参数、写入数据或 DMA 描述符、写命令寄存器、等待完成并检查错误。以磁盘读写为例，驱动程序可能先填写目标逻辑块号、扇区数量和缓冲区地址，然后写入 READ 或 WRITE 命令。设备接到命令后改变状态位，完成后设置就绪位或错误位。

这里的关键是“提交”和“完成”是两个不同时间点。命令寄存器写入成功只表示设备接收了请求，不表示数据已经读到内存或写到介质。驱动必须在完成通知后再唤醒等待线程，并根据错误寄存器判断是否需要重试、上报 EIO 或执行复位。

## 状态位与错误码

常见状态位包括 BUSY、READY、ERROR、DATA REQUEST、FAULT 等。BUSY 表示设备正在处理命令，READY 表示设备可接受新请求，ERROR 表示本次操作失败，DATA REQUEST 表示设备等待主机传输数据。复杂设备还会提供更细的错误原因，如坏块、不可纠正错误、命令中止、介质变更等。

状态位的解释要结合上下文。同一个寄存器地址可能在读时表示状态、写时表示命令；某些错误位只有在 ERROR 置位后才有意义；设备复位后需要等待一段时间才能重新提交请求。驱动程序若只看一个标志而不检查完整状态，容易在边界情况下误判完成或漏报错误。

## 顺序、并发与内存屏障

现代处理器和编译器可能重排普通内存访问，而设备寄存器访问对顺序非常敏感。因此驱动常需要使用 volatile 语义、内存屏障或体系结构提供的 I/O 访问函数，确保先写 DMA 描述符，再通知设备启动；先读取完成状态，再读取设备写入的结果。

如果多个 CPU 或中断处理程序同时访问同一个设备，驱动还必须使用锁保护软件状态，例如请求队列、当前命令、错误计数和等待队列。硬件寄存器协议解决不了内核内部并发，软件必须维护“谁正在提交、谁正在回收、谁可以复位设备”的不变量。

# 轮转调度与时间片选择

## 轮转调度的思想

轮转调度将 CPU 时间切成固定长度的时间片。调度器从就绪队列中选择一个任务运行一个时间片；如果任务没有结束，就把它放回队尾，再运行下一个任务。所有同等级任务依次获得 CPU，因此轮转调度也常被称为时间片调度。

与 FIFO 或 SJF 不同，轮转调度关注交互响应。多个任务同时到达时，每个任务很快都能获得第一次运行机会，不必等前面的任务完整结束。

## 响应时间优势

假设三个任务同时到达且都需要运行较长时间。FIFO 会让第二、第三个任务等待前面任务完成后才能首次运行；轮转调度则可以在几个时间片内让每个任务至少运行一次。对终端输入、图形界面和交互服务来说，这种快速首次响应比单纯追求平均完成时间更重要。

轮转调度还天然避免某个同等级 CPU 密集型任务永久占用处理器。只要时间片到期，定时器中断会把控制权交给内核，调度器可以切换到下一个任务。

## 时间片的取舍

时间片太短，响应时间更好，但上下文切换频率增加。每次切换都要进入内核、保存恢复现场、更新队列，并可能带来缓存和 TLB 失效。如果时间片接近上下文切换成本，CPU 会把大量时间浪费在切换上。

时间片太长，切换成本被摊薄，但交互延迟变大。一个刚到达的任务可能要等当前任务运行很久才有机会执行。实际系统通常根据硬件速度、工作负载和调度类设置毫秒级到几十毫秒级的时间片，并可能对不同优先级使用不同长度。

## 对周转时间的影响

轮转调度不一定优化周转时间。对于一批长度不同的任务，短任务可能被迫与长任务反复轮换，而不是一次性完成。这样短任务的完成时刻可能比 SJF 或 STCF 更晚，平均周转时间变差。

这体现了调度指标冲突：轮转改善响应时间和公平性，但可能牺牲批处理任务的完成效率。桌面和分时系统更容易接受这种代价，纯批处理系统则可能偏好短任务优先类策略。

## 实现细节

轮转调度通常维护一个就绪队列。时间片到期后，如果当前任务仍可运行，就被移到队尾；如果它阻塞等待 I/O，则进入等待队列；如果它退出，则被回收或进入僵尸状态。新到达任务插入队列的位置会影响响应和公平，不同系统可能有不同规则。

时间片必须依赖硬件定时器。调度器不能只相信进程主动让出 CPU，否则轮转无法强制执行。

## 易错点

轮转不是“每个进程同时运行”，而是快速交替运行。时间片也不是越短越好，过短会让上下文切换开销压倒实际工作。分析轮转调度时，应同时看响应时间、上下文切换次数和任务完成时刻。

# CPU 虚拟化：陷入模拟与二进制翻译

## 受限直接执行

虚拟机中的大多数指令可以直接在真实 CPU 上执行。VMM 让客户操作系统和应用运行，只有当它们执行特权操作、发生异常或需要调度切换时，才介入处理。这与操作系统用受限直接执行虚拟化进程类似。

VMM 需要在虚拟机之间做机器切换。切换时保存当前虚拟机的寄存器、程序计数器和特权硬件状态，再恢复另一个虚拟机的状态。相比普通进程上下文切换，它涉及更多硬件控制状态。

## 陷入与模拟

客户操作系统不能真正拥有最高特权级。它执行特权指令时，应陷入 VMM。VMM 检查意图，更新虚拟硬件状态，必要时模拟指令效果，然后让客户系统继续运行。

例如客户 OS 试图设置陷阱向量、修改页表基址或操作中断控制器时，VMM 不能让它直接改真实硬件，而是记录到该虚拟机的虚拟设备状态中。

## 系统调用路径

虚拟机内的用户进程发起系统调用时，逻辑上应进入客户操作系统，而不是由 VMM 处理业务语义。VMM 可能先捕获陷入，再把控制转交给客户 OS 的陷阱处理程序。客户 OS 完成后执行返回特权指令，又可能再次经过 VMM。

这种额外层次会增加系统调用和异常路径开销。硬件辅助虚拟化通过提供新的特权层和 VM entry/exit 机制，使这条路径更直接。

## 可虚拟化条件

理想体系结构中，所有敏感指令都应是特权指令，即客户 OS 执行它们会陷入 VMM。若某些敏感指令在低特权级静默执行但结果不同，VMM 无法自然捕获，就会破坏透明虚拟化。

早期 x86 存在这类问题，因此纯陷入模拟不够。VMM 需要使用二进制翻译或半虚拟化等方法处理。

## 二进制翻译

二进制翻译会扫描客户内核代码，把难以虚拟化的敏感指令替换为安全序列或对 VMM 的调用。普通用户代码仍可直接执行，客户内核中的特殊路径被动态翻译和缓存。

这种技术实现复杂，但曾是高性能 x86 虚拟化的重要基础。现代硬件辅助虚拟化减少了对复杂二进制翻译的依赖，但理解它有助于把握“陷入、模拟、透明性和性能”之间的关系。

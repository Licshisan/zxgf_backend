# 进程控制块 PCB 与内核数据结构

## PCB 的作用

进程控制块通常称为 PCB，是内核用来表示一个进程的核心数据结构。用户程序看到的是 PID、命令行、文件描述符等有限信息；内核需要知道得更多，包括进程处于什么状态、如何恢复执行、使用哪套地址空间、拥有哪些资源、等待哪个事件以及退出后应通知谁。

不同操作系统的结构体名称不同，例如 UNIX 类系统中常见 `task_struct`、`proc` 或类似结构，但职责基本一致：把“运行中的程序”转化为内核可管理的对象。没有 PCB，调度器无法选择进程，内存管理无法找到页表，文件系统无法关联打开文件，父子进程关系也无法维护。

## PCB 中的核心字段

PCB 至少包含进程标识信息，例如 PID、父进程 PID、进程组、会话和所属用户。标识信息用于权限检查、信号投递、命令行工具查询和父子关系管理。它还包含状态字段，记录进程是就绪、运行、阻塞、停止还是僵尸。

执行现场也是 PCB 的关键内容。进程被切换出去时，内核需要保存寄存器、内核栈位置、程序计数器等上下文；再次运行时，这些值用于恢复执行。PCB 还会引用内存管理结构，例如页表根地址、地址空间描述、虚拟内存区域列表。打开文件表、当前工作目录、根目录、信号处理方式、资源限制、优先级、时间片统计等也通常由 PCB 直接保存或间接引用。

## 队列与索引结构

PCB 不只是单独存在，还会被组织进多种内核数据结构。就绪进程通常进入运行队列，调度器从中选择下一个运行对象。等待 I/O 的进程会挂到设备、文件、管道或同步对象对应的等待队列。所有进程还可能通过全局表或哈希表按 PID 索引，方便 `kill(pid)`、`waitpid(pid)` 和系统监控工具定位。

这种多重组织带来一个重要要求：状态和队列必须一致。进程从阻塞变为就绪时，应从等待队列移除并加入运行队列；进程退出时，应从可调度结构中删除，但可能保留一小部分信息给父进程回收。若队列更新遗漏，就可能出现无法唤醒、重复调度或 PID 指向错误对象等严重问题。

## PCB 与上下文切换

上下文切换时，PCB 是保存和恢复状态的落点。内核先把当前进程的寄存器现场写入其 PCB 或内核栈，再把它的状态改为就绪、阻塞或其他合适状态；随后选择另一个进程，从其 PCB 取出地址空间信息和寄存器现场，恢复 CPU 执行。

这里要区分进程上下文切换与普通函数调用。函数调用只在同一执行流内部保存返回地址和少量寄存器；进程切换涉及内核调度、地址空间切换、特权级变化、缓存和 TLB 影响。PCB 中保存的信息越完整，恢复越可靠；但维护字段越多，调度和切换路径也越重。

## 生命周期管理

进程创建时，内核分配并初始化 PCB，设置初始状态、PID、父子关系、地址空间和文件描述符继承关系。运行期间，系统调用、异常和调度事件不断更新 PCB。进程退出时，内核释放地址空间、关闭或减少文件引用、记录退出状态，并通知父进程。父进程调用 `wait()` 后，剩余的僵尸记录才会被彻底清理。

PCB 的生命周期必须防止悬空引用。比如某个进程退出后，如果另一个内核路径仍持有它的指针，就可能访问已释放内存。真实内核通常使用引用计数、锁和明确的状态转换来保护这些结构。

## 易错点

PCB 不是进程的全部内存，也不是用户能随意读取的结构。它是内核侧元数据，描述和连接进程拥有的资源。另一个常见误解是把 PID 当作永久身份。PID 会被复用，只有在结合生命周期、父子关系或创建时间等信息时，才能避免把新进程误认为旧进程。

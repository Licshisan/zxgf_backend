# 休眠锁、两阶段锁与 futex 思想

## 自旋与休眠

自旋锁等待时持续消耗 CPU。若等待时间短，它避免睡眠唤醒开销；若等待时间长，浪费严重。休眠锁在锁不可用时把线程阻塞，交给调度器运行其他线程，等锁释放后再唤醒。

休眠锁适合长临界区、可能阻塞的操作或单 CPU 场景。代价是进入内核、线程阻塞和唤醒的开销比短暂自旋更高。

## 两阶段锁

两阶段锁结合自旋和休眠。线程先自旋一小段时间，期待锁很快释放；如果仍失败，再进入内核睡眠。这样在短等待时避免系统调用，在长等待时避免持续浪费 CPU。

自旋次数或时间阈值很关键。太短会频繁睡眠，太长会浪费 CPU。真实系统可能根据 CPU 数、锁竞争和持锁时间动态调整。

## futex 思想

futex 即 fast userspace mutex，核心思想是在无竞争情况下完全在用户态完成加锁和解锁；只有竞争或需要睡眠唤醒时才进入内核。锁变量保存在用户内存中，用户态用原子操作尝试获取。

若获取失败，线程调用 futex wait，让内核在该用户地址对应的等待队列上睡眠。释放锁时，持有者更新用户态锁变量，并在有等待者时调用 futex wake 唤醒一个或多个线程。

## 优势

大多数锁在无竞争时非常频繁，futex 避免了每次 lock/unlock 都系统调用。竞争时又能让线程睡眠，而不是长期自旋。POSIX 互斥锁等高级同步原语常可基于 futex 实现。

futex 把快速路径留给用户态，把等待队列和阻塞唤醒交给内核，是机制分工清晰的设计。

## 难点

futex 必须处理竞态：线程准备睡眠时，锁可能已经释放。典型 wait 操作会同时检查用户地址值是否仍等于期望值，若已变化就不睡眠，避免丢失唤醒。唤醒也可能是虚假或多余的，调用方必须重新检查条件。

## 易错点

futex 不是一把完整的锁，而是构建锁的底层等待/唤醒机制。两阶段锁也不是总比自旋或休眠好，阈值错误会适得其反。任何休眠等待都必须防止检查条件和进入睡眠之间的丢失唤醒。

# wait 与僵尸进程回收

## wait 的作用

父进程创建子进程后，经常需要知道子进程何时结束以及结束结果是什么。`wait()` 和 `waitpid()` 提供了这个能力：父进程调用后，如果有已退出但尚未回收的子进程，系统调用立即返回；如果没有这样的子进程，但仍有子进程在运行，父进程通常会阻塞，直到某个子进程状态发生变化。

`wait()` 的返回值通常是被回收子进程的 PID，退出状态通过参数传出。父进程可以从状态中判断子进程是正常 `exit()`，还是被信号终止，以及退出码是多少。这对于 shell 显示命令结果、服务器管理工作进程、构建工具判断任务成功失败都很重要。

## 僵尸进程是什么

子进程结束时，内核会释放它的大部分资源，例如用户地址空间、内核栈、打开文件引用等。但内核不能马上删除全部进程记录，因为父进程可能还需要获取退出状态。因此，内核保留一小段记录：PID、退出码、资源使用统计以及父子关系。这种已经结束但尚未被父进程 `wait()` 回收的进程称为僵尸进程。

僵尸进程不再执行代码，也不占用原来的用户内存，但会占用 PID 和少量内核表项。如果父进程长期不回收，大量僵尸会耗尽进程表资源，使系统无法创建新进程。

## waitpid 的精确控制

`wait()` 回收任意一个已结束的子进程；`waitpid()` 可以指定等待某个 PID，或按进程组等待，还可以通过选项实现非阻塞检查。例如 shell 管理后台任务时，不能因为某个后台进程未结束就卡住交互界面，因此会使用带 `WNOHANG` 的等待方式轮询或在收到 `SIGCHLD` 后回收。

精确等待还能避免回收错对象。一个父进程如果创建多个子进程，不同子任务对应不同业务语义，父进程需要知道是哪一个任务完成，才能更新正确的状态。

## 父进程退出后的处理

如果父进程先退出，子进程会成为孤儿进程。传统 UNIX 系统会把孤儿进程重新托管给 `init` 或类似的进程管理器；现代系统中可能由 `systemd` 或子回收器承担该职责。新的父进程会负责在这些进程退出后调用等待接口，防止僵尸永久残留。

这说明僵尸进程并不是“无法杀死的进程”。它已经死了，问题在于退出状态还没有被父进程读取。对僵尸进程发送 `kill` 通常没有意义，真正需要处理的是让父进程回收，或结束有问题的父进程让系统管理进程接管。

## 与同步的关系

`wait()` 同时也是一种父子进程同步机制。父进程在 `wait()` 返回之后，才能确定子进程已经结束。OSTEP 中的简单例子展示了这一点：如果父进程在 `fork()` 后立即打印，父子输出顺序不确定；如果父进程先 `wait()`，那么子进程结束前父进程不会继续执行后续打印。

不过 `wait()` 只能同步“子进程结束”这个事件，不能表达任意中间条件。若父进程需要知道子进程已经完成初始化、写入某段数据或准备好接收请求，应使用管道、信号、共享内存加同步原语等更合适的 IPC 机制。

## 易错点

僵尸进程不是阻塞进程，也不是后台运行进程。阻塞进程还活着，等待事件后可继续运行；后台进程也可能正常执行；僵尸进程已经结束，只剩退出记录。另一个误区是认为父进程 `wait()` 只是“暂停一下”。实际上它改变了子进程资源回收状态，是进程生命周期管理的一部分。

# exec 与程序映像替换

## exec 的基本含义

`exec` 不是创建新进程，而是在当前进程中装入并运行另一个程序。调用成功后，当前进程的用户态程序映像被替换：原来的代码段、数据段、堆、栈和大部分用户态执行状态不再存在，新程序从自己的入口点开始执行。进程的 PID 通常保持不变，因此从外部看是同一个进程换了一个程序在运行。

UNIX 提供多种 `exec` 变体，例如 `execl`、`execv`、`execvp` 等，差别主要在参数传递形式以及是否按 `PATH` 查找可执行文件。它们的核心语义相同：成功后不返回；只有失败时才返回到调用点，并设置错误原因。

## 程序映像如何替换

执行 `exec` 时，内核首先检查可执行文件是否存在、权限是否允许、格式是否可识别。随后根据可执行文件格式建立新的地址空间布局，映射代码段、只读数据段、可写数据段，创建新的堆和用户栈。命令行参数和环境变量被复制到新栈上，使新程序能通过 `argc`、`argv` 和环境变量读取启动信息。

动态链接程序还需要加载解释器或动态链接器，由它继续装入共享库并完成符号重定位。脚本文件可能通过 shebang 行指定解释器，例如 `#!/bin/sh`。这些细节都发生在“当前进程变成另一个程序”的语义之下。

## 哪些资源会保留

`exec` 会替换用户态内存，但不会简单清空整个进程身份。PID、父进程关系、进程组、当前工作目录、资源限制等通常保留。文件描述符默认也会保留，除非它们设置了 close-on-exec 标志。这个特性非常关键：shell 可以在 `exec` 前打开文件、复制描述符或连接管道，让新程序继承已经准备好的标准输入输出。

信号处理在 `exec` 后会有特殊规则：用户自定义处理函数通常会重置，因为旧函数地址属于已被替换的程序映像；被忽略的信号可能继续保持忽略。不同系统有细节差异，但原则是不能让新程序跳转到旧地址空间中的处理代码。

## fork 与 exec 的组合

单独使用 `fork()` 会得到当前程序的副本，单独使用 `exec()` 会替换当前进程。二者组合后，父进程可以保留原程序继续控制流程，子进程则替换成目标程序。这正是 shell 启动外部命令的基本模型。

例如执行 `ls > out.txt` 时，shell 先 `fork()`。子进程关闭标准输出，打开 `out.txt`，使文件描述符 1 指向该文件，然后 `exec()` 执行 `ls`。`ls` 并不知道自己被重定向，只是照常向标准输出写数据；父进程等待或继续处理下一条命令。

## 错误处理

因为成功的 `exec` 不返回，所以 `exec` 调用后的代码通常只处理失败路径。常见失败原因包括文件不存在、没有执行权限、参数列表过长、可执行格式错误、内存不足或路径查找失败。健壮程序在子进程中调用 `exec` 失败后应输出错误并退出，避免子进程误继续执行父进程逻辑。

在多进程程序中，这一点尤其重要。如果 shell 的子进程 `exec` 失败却没有退出，它可能回到 shell 主循环，导致两个 shell 同时读取终端输入，行为会非常混乱。

## 易错点

`exec` 后 PID 不变，但程序内容变了；`fork` 后程序内容相同，但 PID 不同。`exec` 不是“调用另一个程序然后回来”，而是用另一个程序取代自己。还要注意文件描述符继承：它既是实现重定向和管道的基础，也是资源泄漏的来源。服务程序启动子程序时，常需要设置 close-on-exec，避免敏感描述符被意外继承。

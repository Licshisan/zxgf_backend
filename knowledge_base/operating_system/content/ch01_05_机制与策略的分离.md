# 机制与策略的分离

## 核心定义
机制回答“系统如何做”，策略回答“系统选择做什么”。在操作系统中，机制通常是底层能力或执行路径，例如上下文切换、陷入处理、页表地址转换、锁的获取与释放、磁盘块读写。策略则是在多个可选对象或方案之间做决定，例如选择哪个进程运行、淘汰哪个页面、分配哪个空闲块、按什么顺序处理磁盘请求。

机制与策略分离是一条重要设计原则。它让系统能在不重写底层能力的情况下更换决策算法，也让同一机制服务多种应用场景。

## 为什么要分离
操作系统既要可靠执行底层操作，又要根据工作负载做出选择。如果把机制和策略混在一起，任何策略调整都可能影响底层正确性。例如上下文切换本身只负责保存旧进程寄存器、恢复新进程寄存器；至于新进程是谁，应由调度策略决定。

分离后，系统更容易维护和扩展。调度器可以从 FIFO 改为 MLFQ 或比例份额调度，而上下文切换机制仍然可用。页面替换策略可以从 FIFO 改为近似 LRU，而页表、缺页异常和换入换出机制仍然保持基本结构。

## 机制的特点
机制通常要求正确、稳定、低开销。它直接处理硬件状态和核心数据结构，一旦出错会造成严重后果。陷入机制必须保证用户态只能从合法入口进入内核；地址转换机制必须保证页表权限被正确执行；锁机制必须保证临界区互斥；日志提交机制必须保证恢复时能识别完整事务。

机制不应该嵌入过多场景假设。一个上下文切换机制不应只适用于某一种调度算法；一个页表机制也不应只服务某一种替换策略。机制越通用，越能支撑不同策略。

## 策略的特点
策略通常体现目标和权衡。调度策略可能追求低响应时间、短平均周转时间、公平性或实时性；内存替换策略可能追求低缺页率、低实现开销或避免抖动；磁盘调度策略可能追求少寻道、高吞吐或请求公平。

策略没有脱离场景的绝对最优。SJF 在已知作业长度时能降低平均周转时间，但真实系统往往不知道未来运行时间；LRU 利用局部性，但精确实现成本高；NFS 的无状态设计便于恢复，却牺牲某些一致性语义。

## 典型例子
CPU 调度是最清晰的例子。上下文切换、定时器中断和就绪队列操作属于机制；选择 FIFO、轮转、MLFQ、彩票调度或多处理器负载均衡属于策略。机制保证“能切换”，策略决定“切给谁”。

虚拟内存中，缺页异常、页表修改、磁盘换入换出属于机制；页面替换算法属于策略。文件系统中，块读写和元数据更新属于机制；空闲块分配、缓存写回时机、日志模式则包含策略选择。

## 分离的边界
机制和策略并非总能完全分开。策略可能需要机制提供信息，例如调度策略需要进程优先级、运行时间和阻塞历史；页面替换策略需要引用位、脏位或访问时间近似信息。机制也可能因策略需求而设计额外字段或快速路径。

关键是避免把具体决策硬编码在底层执行路径中。机制应提供足够信息和操作能力，策略在清晰边界上做决定。

## 易错点
常见错误是把某个算法误称为机制。例如“LRU 页面替换”不是换页机制本身，而是决定淘汰页面的策略。另一个错误是认为机制与策略分离只是代码结构问题。它更是系统设计问题，关系到可维护性、可扩展性和可验证性。

还要注意策略也需要正确性约束。调度策略可以不同，但不能选择不存在的进程；页面替换策略可以不同，但不能淘汰被固定或正在 I/O 的页面；日志策略可以不同，但不能破坏崩溃恢复的不变量。

## 与其他知识点的关系
后续几乎所有章节都会反复出现这组概念。受限直接执行讲 CPU 虚拟化机制，调度章节讲 CPU 分配策略；分页章节讲地址转换机制，超越物理内存讲替换策略；文件系统实现讲布局机制，崩溃一致性讲写入顺序和日志策略。掌握这一区分能避免把“怎么做到”和“为什么这样选”混在一起。

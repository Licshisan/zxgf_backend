# 上下文切换中的 TLB 刷新与 ASID

## 为什么切换会影响 TLB

TLB 缓存的是虚拟页号到物理页框号的映射，而不同进程有不同页表。同一个 VPN 在进程 A 中可能映射到 PFN 10，在进程 B 中映射到 PFN 200。若上下文切换后继续使用旧 TLB 项，进程 B 可能访问到进程 A 的物理页，严重破坏隔离。

因此，切换地址空间时必须处理 TLB 中属于旧进程的映射。最简单方法是刷新 TLB，使旧转换失效。

## 全量刷新

全量刷新会清空或失效所有非全局 TLB 项。它简单可靠，但有性能代价。新进程刚运行时，TLB 为空，需要逐步重新填充，初期内存访问会产生更多未命中。

如果进程频繁切换，TLB 刷新成本会很明显。上下文切换本身已经有寄存器保存、调度和缓存影响，TLB 冷启动会进一步增加开销。

## ASID

ASID 是地址空间标识符。TLB 项除了保存 VPN 和 PFN，还保存所属地址空间的 ASID。硬件查 TLB 时同时比较 VPN 和 ASID，只有二者都匹配才命中。这样不同进程相同 VPN 的映射可以同时留在 TLB 中。

上下文切换时，内核只需把当前 ASID 切换为新进程的 ASID，不必清空整个 TLB。若进程切回，之前的 TLB 项可能仍然有效，命中率更好。

## ASID 复用

ASID 数量有限。系统进程数可能超过可用 ASID，因此操作系统需要复用 ASID。复用前必须确保旧 ASID 对应的 TLB 项不再被误用，通常需要按 ASID 刷新或维护代际信息。

某些映射可标记为全局，例如内核在所有进程中相同的映射。全局 TLB 项不随普通地址空间切换失效，减少系统调用和中断进入内核时的转换开销。

## 多核中的 TLB 一致性

多处理器系统中，每个 CPU 通常有自己的 TLB。若一个 CPU 修改了某进程页表，其他正在运行同一地址空间的 CPU 可能仍缓存旧映射。操作系统需要发送 TLB shootdown，让相关 CPU 失效旧项。

这类跨核失效成本较高，涉及中断、同步和等待。因此内核会尽量批量处理页表更新，减少不必要的 shootdown。

## 易错点

刷新 TLB 不是刷新数据缓存。它只影响地址转换缓存。ASID 也不是 PID，虽然常与进程相关，但它是硬件可识别的地址空间标签，数量和生命周期由操作系统管理。
